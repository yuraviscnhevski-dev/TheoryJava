## 1. Spring Security: что такое авторизация и аутентификация?

Это два фундаментальных понятия безопасности, которые часто путают:

| Термин | Определение | Пример |
|--------|-------------|--------|
| **Аутентификация** | Процесс проверки, кто ты такой. Подтверждение личности пользователя | Ввод логина и пароля, проверка по отпечатку пальца |
| **Авторизация** | Процесс проверки, что тебе разрешено делать. Определение прав доступа | Доступ к странице `/admin` только для роли ADMIN |

**Простыми словами:**
- Аутентификация отвечает на вопрос **"Кто ты?"**
- Авторизация отвечает на вопрос **"Что тебе можно?"**

В Spring Security эти процессы разделены и настраиваются отдельно. Аутентификация происходит через `AuthenticationManager`, а авторизация — через конфигурацию `HttpSecurity` [ссылка на источник].

## 2. Объекты Principal, Authorities, Authentication

Это ключевые интерфейсы Spring Security, которые нужно понимать:

| Интерфейс | Назначение | Ключевые методы |
|-----------|------------|-----------------|
| **Principal** | Базовое представление пользователя (из Java SE) | `getName()` |
| **Authentication** | Основной объект аутентификации в Spring, расширяет Principal | `getPrincipal()`, `getCredentials()`, `getAuthorities()`, `isAuthenticated()` |
| **GrantedAuthority** | Представляет разрешение/роль пользователя | `getAuthority()` |

**Подробнее об Authentication:**

`Authentication` — это центральный интерфейс, который представляет:
- **Запрос на аутентификацию** (еще не проверенный пользователь)
- **Аутентифицированного пользователя** (уже проверенный)

Основные методы:
- `getPrincipal()` — возвращает идентификатор пользователя (обычно объект `UserDetails`)
- `getCredentials()` — возвращает учетные данные (обычно пароль, после аутентификации очищается)
- `getAuthorities()` — возвращает коллекцию разрешений пользователя
- `isAuthenticated()` — показывает, прошел ли пользователь аутентификацию

**Пример иерархии:**
SecurityContextHolder → SecurityContext → Authentication → Principal + Authorities

text

После успешной аутентификации объект `Authentication` помещается в `SecurityContext`, который хранится в `SecurityContextHolder`.

## 3. Чем отличается InMemoryAuthentication от BasicAuthentication?

Это вообще разные вещи, их часто путают. Давай разберем:

| Критерий | InMemoryAuthentication | BasicAuthentication |
|----------|------------------------|---------------------|
| **Что это** | Способ **хранения** пользователей | Способ **передачи** учетных данных |
| **Суть** | Хранение учетных записей прямо в памяти приложения | HTTP-аутентификация через заголовок `Authorization` |
| **Где хранятся данные** | В памяти (оперативная память) | Не хранятся, передаются с каждым запросом |
| **Когда использовать** | Тестирование, прототипы, небольшие приложения | API, сервисы, stateless-аутентификация |

**InMemoryAuthentication — пример:**
```java
@Configuration
public class SecurityConfig {
    
    @Bean
    public UserDetailsService users() {
        UserDetails user = User.builder()
            .username("user")
            .password("{noop}password")
            .roles("USER")
            .build();
        return new InMemoryUserDetailsManager(user);
    }
}
BasicAuthentication — пример:

java
@Configuration
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .anyRequest().authenticated()
                .and()
            .httpBasic(); // Включаем Basic-аутентификацию
        return http.build();
    }
}
Важно: InMemoryAuthentication можно комбинировать с BasicAuthentication — хранить пользователей в памяти, но передавать учетные данные через Basic-заголовок.

4. Как мы можем добавить секьюрность к контроллеру? (минимум 2 способа)
Есть несколько способов, вот основные:

Способ 1: Глобальная конфигурация через HttpSecurity
java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .antMatchers("/public/**").permitAll()  // Доступно всем
                .antMatchers("/users/**").hasRole("USER")  // Только USER
                .antMatchers("/admin/**").hasRole("ADMIN")  // Только ADMIN
                .anyRequest().authenticated()  // Все остальные запросы требуют аутентификации
                .and()
            .httpBasic();
        
        return http.build();
    }
}
Способ 2: Аннотации на уровне методов
java
@RestController
@RequestMapping("/api/users")
public class UserController {
    
    @GetMapping("/profile")
    @PreAuthorize("isAuthenticated()")  // Только аутентифицированные
    public User getProfile() {
        return userService.getCurrentUser();
    }
    
    @GetMapping
    @PreAuthorize("hasRole('ADMIN')")  // Только ADMIN
    public List<User> getAllUsers() {
        return userService.findAll();
    }
    
    @PostMapping
    @Secured("ROLE_ADMIN")  // Альтернативная аннотация
    public User createUser(@RequestBody User user) {
        return userService.save(user);
    }
}
Для работы аннотаций нужно включить:

java
@Configuration
@EnableGlobalMethodSecurity(prePostEnabled = true, securedEnabled = true)
public class MethodSecurityConfig {
    // ...
}
Способ 3: Фильтры (продвинутый уровень)
Можно создать свои фильтры, которые будут перехватывать запросы до контроллера.

5. Связи таблиц many-to-many, one-to-one
В JPA/Hibernate эти связи настраиваются аннотациями. Понимание важно и для Hibernate, и для проектирования БД.

One-to-One (Один к одному)
Каждой записи в одной таблице соответствует ровно одна запись в другой.

Пример: пользователь и паспорт

java
@Entity
public class User {
    @Id
    @GeneratedValue
    private Long id;
    
    @OneToOne(mappedBy = "user")
    private Passport passport;
}

@Entity
public class Passport {
    @Id
    @GeneratedValue
    private Long id;
    
    @OneToOne
    @JoinColumn(name = "user_id")
    private User user;
}
В БД: обычно внешний ключ на стороне "владельца" (здесь в passport есть user_id).

Many-to-One и One-to-Many (Многие к одному и Один ко многим)
Самые частые связи. Например, много заказов у одного пользователя.

java
@Entity
public class User {
    @Id
    @GeneratedValue
    private Long id;
    
    @OneToMany(mappedBy = "user")
    private List<Order> orders;
}

@Entity
public class Order {
    @Id
    @GeneratedValue
    private Long id;
    
    @ManyToOne
    @JoinColumn(name = "user_id")
    private User user;
}
Many-to-Many (Многие ко многим)
Например, студенты и курсы: один студент может учиться на многих курсах, на одном курсе много студентов.

Вариант 1: Простая связь через @ManyToMany

java
@Entity
public class Student {
    @Id
    @GeneratedValue
    private Long id;
    
    @ManyToMany
    @JoinTable(
        name = "student_course",
        joinColumns = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private List<Course> courses;
}

@Entity
public class Course {
    @Id
    @GeneratedValue
    private Long id;
    
    @ManyToMany(mappedBy = "courses")
    private List<Student> students;
}
Вариант 2: Связующая сущность (рекомендуется, если нужно хранить доп. данные)

java
@Entity
public class StudentCourse {
    @Id
    @GeneratedValue
    private Long id;
    
    @ManyToOne
    @JoinColumn(name = "student_id")
    private Student student;
    
    @ManyToOne
    @JoinColumn(name = "course_id")
    private Course course;
    
    private LocalDate enrollmentDate;  // Дополнительное поле
}

6. Как работают каскады для таблиц и какие они бывают?
Каскады определяют, как операции над родительской сущностью влияют на связанные дочерние.

Типы каскадов в JPA/Hibernate:
Тип	Описание
ALL	Все операции каскадируются (persist, merge, remove, refresh, detach)
PERSIST	При сохранении родителя сохраняются и дети
MERGE	При обновлении родителя обновляются и дети
REMOVE	При удалении родителя удаляются и дети
REFRESH	При обновлении состояния родителя обновляются и дети
DETACH	При отсоединении родителя отсоединяются и дети
Пример использования:
java
@Entity
public class User {
    @Id
    @GeneratedValue
    private Long id;
    
    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL)
    private List<Order> orders;
}

// Теперь при сохранении User:
User user = new User();
Order order = new Order();
user.setOrders(List.of(order));

userRepository.save(user);  // order сохранится автоматически!
Каскады на уровне БД (отличаются от JPA!):
В самой базе данных тоже есть каскады для внешних ключей:

ON DELETE CASCADE — при удалении родителя удаляются дети

ON UPDATE CASCADE — при обновлении ключа родителя обновляются ключи детей

8. REST-сервисы. Их преимущества и недостатки
REST (Representational State Transfer) — архитектурный стиль для создания веб-сервисов, основанный на принципах HTTP.

Ключевые принципы REST:
Клиент-сервер — разделение интерфейса и хранилища данных

Отсутствие состояния (stateless) — каждый запрос содержит всю информацию для его обработки

Кэширование — ответы могут быть помечены как кэшируемые или нет

Единообразие интерфейса — единые принципы работы с ресурсами

Слои — возможность использования промежуточных серверов

Код по требованию (опционально) — возможность передачи исполняемого кода клиенту

Преимущества REST :
✅ Простота — использует стандартные HTTP-методы

✅ Масштабируемость — stateless позволяет легко масштабировать

✅ Гибкость — разные форматы данных (JSON, XML, HTML)

✅ Кэширование — повышает производительность

✅ Универсальность — подходит для разных клиентов (web, mobile, IoT)

Недостатки и ограничения :
❌ Избыточность данных — часто возвращает больше данных, чем нужно

❌ Сложность с связанными данными — может потребоваться несколько запросов

❌ Отсутствие строгой спецификации — многое остается на усмотрение разработчика

❌ Перегрузка — иногда сложно получить связанные данные за один запрос

⬆️ К оглавлению

9. Форматы данных, использующиеся в REST-сервисах
Основные форматы обмена данными в REST:

Формат	Особенности	Пример
JSON	Самый популярный, легкий, человекочитаемый	{"name": "Иван", "age": 25}
XML	Более тяжелый, строгая структура, устаревает	<person><name>Иван</name><age>25</age></person>
YAML	Читаемый, часто для конфигов	name: Иван age: 25
HTML	Для веб-страниц	<h1>Иван</h1>
Plain text	Простой текст	Иван, 25
Binary	Картинки, файлы	JPEG, PNG, PDF
Как задается формат:
Через HTTP-заголовки:

Accept — что клиент хочет получить

Content-Type — что клиент отправляет

http
GET /api/users/1
Accept: application/json

POST /api/users
Content-Type: application/json

10. Что такое responseBody, requestBody, ResponseEntity?
Это аннотации и классы Spring для работы с HTTP.

@RequestBody
Что делает: преобразует тело HTTP-запроса в Java-объект

Где ставить: на параметр метода контроллера

Как работает: Spring автоматически десериализует JSON/XML в объект

java
@PostMapping("/users")
public User createUser(@RequestBody User user) {  // ← JSON из запроса → объект User
    return userService.save(user);
}
@ResponseBody
Что делает: преобразует Java-объект в тело HTTP-ответа

Где ставить: на метод контроллера или класс

Как работает: Spring автоматически сериализует объект в JSON/XML

java
@GetMapping("/users/{id}")
@ResponseBody
public User getUser(@PathVariable Long id) {
    return userService.findById(id);
}
ResponseEntity
Что это: полное управление HTTP-ответом (тело, статус, заголовки)

Когда использовать: когда нужен контроль над статусом и заголовками

java
@GetMapping("/users/{id}")
public ResponseEntity<User> getUser(@PathVariable Long id) {
    User user = userService.findById(id);
    if (user == null) {
        return ResponseEntity.notFound().build();  // 404 Not Found
    }
    return ResponseEntity.ok(user);  // 200 OK с телом
}
Сравнение:
Возвращаемый тип	Контроль статуса	Контроль заголовков	Тело ответа
@ResponseBody	❌ Нет (только 200)	❌ Нет	✅ Есть
ResponseEntity	✅ Полный	✅ Полный	✅ Есть

11. Что такое AJAX/fetch?
AJAX (Asynchronous JavaScript and XML)
Технология для асинхронного обмена данными с сервером

Позволяет обновлять часть страницы без перезагрузки

Исторически использовал XMLHttpRequest

Fetch API
Современная замена AJAX (XMLHttpRequest)

Встроена в современные браузеры

Использует Promise, более удобный синтаксис

Пример fetch :
javascript
// GET-запрос
fetch('https://api.example.com/users/1')
    .then(response => response.json())
    .then(data => console.log(data))
    .catch(error => console.error('Ошибка:', error));

// POST-запрос с JSON
fetch('https://api.example.com/users', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json'
    },
    body: JSON.stringify({name: 'Иван', age: 25})
})
.then(response => response.json())
.then(data => console.log(data));
Основные методы fetch :
get() — GET-запрос

post(data) — POST-запрос с телом

put(data) — PUT-запрос с телом

delete() — DELETE-запрос

abort() — отмена запроса

12. Чем аннотация RestController отличается от Controller?
Это важное различие в Spring MVC.

Аннотация	@Controller	@RestController
Что это	MVC-контроллер	@Controller + @ResponseBody на всех методах
Возвращаемое значение	Имя представления (view)	Объект → JSON/XML
Использование	Веб-страницы (HTML, JSP, Thymeleaf)	REST API
Нужна ли @ResponseBody	Да, на каждом методе	Нет, автоматически
Пример @Controller:
java
@Controller
public class WebController {
    
    @GetMapping("/hello")
    public String hello(Model model) {
        model.addAttribute("message", "Hello!");
        return "hello";  // Возвращает имя представления
    }
}
Пример @RestController :
java
@RestController
public class ApiController {
    
    @GetMapping("/api/hello")
    public Message hello() {
        return new Message("Hello!");  // Возвращает объект → JSON
    }
}
Как работает :
@Controller ожидает, что метод вернет имя представления

@RestController автоматически сериализует возвращаемый объект в JSON

@RestController = @Controller + @ResponseBody на всех методах

13. RestTemplate и его методы
RestTemplate — синхронный HTTP-клиент в Spring для вызова REST-сервисов (устаревает в пользу WebClient, но все еще часто используется).

Основные методы RestTemplate:
Метод	Описание	HTTP-метод
getForObject()	GET-запрос, возвращает объект	GET
getForEntity()	GET-запрос, возвращает ResponseEntity	GET
postForObject()	POST-запрос, возвращает объект	POST
postForEntity()	POST-запрос, возвращает ResponseEntity	POST
put()	PUT-запрос	PUT
delete()	DELETE-запрос	DELETE
exchange()	Полный контроль над запросом	Любой
execute()	Низкоуровневое выполнение	Любой
Примеры использования:
java
@Service
public class UserServiceClient {
    
    private final RestTemplate restTemplate;
    
    public UserServiceClient() {
        this.restTemplate = new RestTemplate();
    }
    
    // getForObject — получаем объект
    public User getUser(Long id) {
        String url = "https://api.example.com/users/{id}";
        return restTemplate.getForObject(url, User.class, id);
    }
    
    // getForEntity — получаем ResponseEntity со статусом
    public ResponseEntity<User> getUserWithStatus(Long id) {
        String url = "https://api.example.com/users/{id}";
        return restTemplate.getForEntity(url, User.class, id);
    }
    
    // postForObject — создаем ресурс
    public User createUser(User user) {
        String url = "https://api.example.com/users";
        return restTemplate.postForObject(url, user, User.class);
    }
    
    // exchange — полный контроль
    public User getUserWithHeaders(Long id) {
        String url = "https://api.example.com/users/{id}";
        
        HttpHeaders headers = new HttpHeaders();
        headers.set("Authorization", "Bearer token");
        HttpEntity<?> entity = new HttpEntity<>(headers);
        
        ResponseEntity<User> response = restTemplate.exchange(
            url, HttpMethod.GET, entity, User.class, id);
        
        return response.getBody();
    }
    
    // put — обновление
    public void updateUser(Long id, User user) {
        String url = "https://api.example.com/users/{id}";
        restTemplate.put(url, user, id);
    }
    
    // delete — удаление
    public void deleteUser(Long id) {
        String url = "https://api.example.com/users/{id}";
        restTemplate.delete(url, id);
    }
}

14. HTTP протокол
HTTP (Hypertext Transfer Protocol) — протокол прикладного уровня для передачи гипертекста.

Основные свойства :
Клиент-серверная архитектура — клиенты инициируют запросы, серверы отвечают

Stateless (без состояния) — каждый запрос независим, сервер не хранит состояние между запросами

Работает поверх TCP (обычно порт 80, HTTPS — 443)

Структура HTTP-сообщения :
Каждое HTTP-сообщение состоит из трех частей:

Стартовая строка (запрос или ответ)

Заголовки (Headers) — метаданные

Тело (Body) — данные (может отсутствовать)

Методы HTTP-запросов :
Метод	Описание	Идемпотентность	Безопасность
GET	Получение ресурса	✅ Да	✅ Да
POST	Создание ресурса	❌ Нет	❌ Нет
PUT	Полное обновление ресурса	✅ Да	❌ Нет
PATCH	Частичное обновление	❌ Нет	❌ Нет
DELETE	Удаление ресурса	✅ Да	❌ Нет
HEAD	GET без тела	✅ Да	✅ Да
OPTIONS	Получение поддерживаемых методов	✅ Да	✅ Да
Коды ответов HTTP :
Диапазон	Категория	Примеры
1xx	Информационные	100 Continue
2xx	Успех	200 OK, 201 Created, 204 No Content
3xx	Перенаправление	301 Moved Permanently, 302 Found
4xx	Ошибка клиента	400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found
5xx	Ошибка сервера	500 Internal Server Error, 503 Service Unavailable
Заголовки HTTP:
Заголовки запроса:

Host — обязательный заголовок с именем сервера

User-Agent — информация о клиенте

Accept — поддерживаемые форматы ответа

Authorization — учетные данные

Заголовки ответа:

Content-Type — формат тела ответа

Content-Length — размер тела

Set-Cookie — установка cookie

Cache-Control — управление кэшированием

Пример HTTP-запроса:
text
GET /api/users/123 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Basic dXNlcjpwYXNz
Пример HTTP-ответа:
text
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 85

{"id":123,"name":"Иван","email":"ivan@example.com"}
