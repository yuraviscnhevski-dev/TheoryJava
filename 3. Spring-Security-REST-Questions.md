## 1. Что такое Spring Security? [вопросы](#вопросы)

 Spring Security — это мощный и высоконастраиваемый фреймворк аутентификации и авторизации, являющийся стандартом де-факто для защиты Spring-приложений. Он обеспечивает комплексную защиту от множества атак и интегрируется с различными протоколами.

🔎 Архитектура и ключевые компоненты
Spring Security построен на цепочке сервлетных фильтров (FilterChainProxy). Каждый фильтр отвечает за одну функцию безопасности (например, аутентификация, CSRF, обработка исключений). Фильтры регистрируются в контейнере сервлетов и обрабатывают входящие запросы в определённом порядке.

Основные компоненты:

SecurityContextHolder — хранилище SecurityContext (по умолчанию используется ThreadLocal), содержащего объект Authentication текущего пользователя. Это обеспечивает доступ к информации о пользователе на протяжении всего запроса.

Authentication — интерфейс, представляющий запрос на аутентификацию или уже аутентифицированного пользователя. Содержит:

principal — идентификатор пользователя (обычно UserDetails).

credentials — учётные данные (пароль, токен), которые часто очищаются после успешной аутентификации.

authorities — коллекция GrantedAuthority.

AuthenticationManager — главный стратегический интерфейс для обработки аутентификации. Единственный метод authenticate(Authentication). Основная реализация — ProviderManager, который делегирует проверку одному или нескольким AuthenticationProvider.

AuthenticationProvider — реализует конкретную логику аутентификации (например, по базе данных, LDAP, JWT). Каждый провайдер поддерживает определённый тип Authentication.

UserDetailsService — загружает данные пользователя по имени пользователя. Возвращает объект UserDetails (содержит имя, пароль, права). Используется стандартными провайдерами, например DaoAuthenticationProvider.

GrantedAuthority — представляет права/роли пользователя. Обычно это строки, например ROLE_USER, READ_PRIVILEGE. Spring Security автоматически добавляет префикс ROLE_ при использовании методов hasRole().

🛠 Пример настройки с несколькими провайдерами
java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .antMatchers("/public/**").permitAll()
                .antMatchers("/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            .and()
            .formLogin()
            .and()
            .httpBasic();
        return http.build();
    }

    @Bean
    public AuthenticationManager authenticationManager(
            UserDetailsService userDetailsService,
            PasswordEncoder passwordEncoder) {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider();
        provider.setUserDetailsService(userDetailsService);
        provider.setPasswordEncoder(passwordEncoder);

        return new ProviderManager(provider); // можно добавить другие провайдеры
    }
}
🔐 Защита от атак
CSRF (Cross-Site Request Forgery) — по умолчанию включён для состояний, требующих сессию. Для stateless REST API рекомендуется отключать: http.csrf().disable().

Session Fixation — защита путём смены идентификатора сессии после аутентификации (настраивается через sessionManagement().sessionFixation().migrateSession()).

CORS (Cross-Origin Resource Sharing) — настройка через http.cors() и @CrossOrigin.



## 2. Что такое аутентификация и авторизация? [вопросы](#вопросы)
Это два фундаментальных процесса безопасности, которые часто путают.

Термин	Определение	Аналогия
Аутентификация	Процесс проверки подлинности субъекта: подтверждение, что он тот, за кого себя выдаёт.	Предъявление паспорта на входе.
Авторизация	Процесс проверки, имеет ли аутентифицированный субъект право выполнять действие или получать ресурс.	Проверка, есть ли у вас пропуск в определённую зону.
🔎 Детали в Spring Security
Аутентификация может быть выполнена множеством способов:

HTTP Basic — передача логина/пароля в заголовке.

Form Login — HTML-форма.

JWT (JSON Web Token) — токен в заголовке Authorization.

OAuth2 / OpenID Connect — делегированная аутентификация.

Remember-Me — долгосрочная аутентификация через куки.

После успешной аутентификации создаётся объект Authentication и помещается в SecurityContextHolder. Далее каждый запрос проходит через фильтры, которые восстанавливают контекст (например, SecurityContextPersistenceFilter).

Авторизация в Spring Security реализуется на двух уровнях:

Уровень URL — через конфигурацию HttpSecurity (например, .antMatchers("/admin/**").hasRole("ADMIN")).

Уровень методов — через аннотации (@PreAuthorize, @Secured, @RolesAllowed). Для этого требуется @EnableGlobalMethodSecurity.

🧑‍💻 Пример авторизации через аннотацию
java

@RestController
public class UserController {
    @PreAuthorize("hasRole('ADMIN') or #id == authentication.principal.id")
    @GetMapping("/users/{id}")
    public User getUser(@PathVariable Long id) { ... }
}

Здесь использован SpEL (Spring Expression Language), позволяющий писать сложные логические выражения.



## 3. Объекты Principal, Authorities, Authentication [вопросы](#вопросы)
В Spring Security эти интерфейсы являются ключевыми.

🔎 Подробное описание
Principal — интерфейс из пакета java.security. Представляет абстрактное понятие субъекта (пользователя, устройства и т.д.). В Spring Security обычно реализуется через UserDetails или просто строку с именем.

Authentication — центральный интерфейс, расширяющий Principal. Содержит:

getPrincipal() — возвращает объект пользователя (чаще всего UserDetails).

getCredentials() — учётные данные (пароль, токен). После аутентификации их часто стирают для безопасности.

getAuthorities() — коллекция GrantedAuthority.

isAuthenticated() — флаг, прошла ли аутентификация.

setAuthenticated(boolean) — используется для изменения статуса.

GrantedAuthority — интерфейс, представляющий право. Метод getAuthority() возвращает строку. В Spring Security есть класс SimpleGrantedAuthority.

🛠 Как работает получение текущего пользователя
java
@GetMapping("/profile")
public String profile() {
    Authentication auth = SecurityContextHolder.getContext().getAuthentication();
    String username = auth.getName(); // получаем имя
    Collection<? extends GrantedAuthority> authorities = auth.getAuthorities();
    return "User: " + username + ", roles: " + authorities;
}
Варианты внедрения Authentication в контроллер:

Через параметр Authentication (Spring автоматически подставит).

Через @AuthenticationPrincipal UserDetails user (более типобезопасно).

Через SecurityContextHolder напрямую (менее элегантно).

🔐 Анонимная аутентификация
Если пользователь не аутентифицирован, Spring Security создаёт специальный объект AnonymousAuthenticationToken с ролью ROLE_ANONYMOUS. Это позволяет настраивать доступ для анонимных пользователей.

## 4. Чем отличается InMemoryAuthentication от basicAuthentication? [вопросы](#вопросы)
Это сравнение не совсем корректно, так как это понятия из разных категорий.

InMemoryAuthentication — это способ хранения пользователей. Позволяет задать список пользователей прямо в коде (в памяти). Используется для тестов и простых приложений.

HTTP Basic Authentication — это механизм аутентификации, при котором логин и пароль передаются в заголовке Authorization в закодированном виде (Base64). Это способ передачи данных, а не их хранения.

Критически важно понимать: это ортогональные понятия, их нельзя противопоставлять. Ментор проверяет, понимаете ли вы уровни абстракции.

Аспект	InMemoryAuthentication	BasicAuthentication
Что это?	Способ хранения учётных записей пользователей (UserDetailsService), реализованный в памяти.	Механизм (схема) аутентификации по протоколу HTTP, при котором логин и пароль передаются в заголовке Authorization: Basic base64(login:password).
Где настраивается?	В UserDetailsService (например, InMemoryUserDetailsManager) или через AuthenticationManagerBuilder (в старых версиях).	В HttpSecurity через метод .httpBasic().
Область применения	Подходит для прототипирования, тестов, небольших приложений без БД.	Подходит для REST API, где клиент может отправлять базовую аутентификацию с каждым запросом (например, для интеграции).
Комбинирование	Можно использовать вместе с basicAuthentication (хранить пользователей в памяти, а аутентифицировать через Basic).	Можно использовать с любым хранилищем (in-memory, JDBC, LDAP).
✅ Пример совместного использования
java
@Bean
public UserDetailsService users() {
    UserDetails user = User.withUsername("user")
            .password("{noop}password")
            .roles("USER")
            .build();
    return new InMemoryUserDetailsManager(user);
}

// В SecurityFilterChain
http.httpBasic(); // включает Basic-аутентификацию
⚠️ Важно про безопасность Basic
Basic-аутентификация передаёт учётные данные в открытом виде (base64 — не шифрование). Обязательно использовать HTTPS.

Для повышения безопасности можно комбинировать с Digest-аутентификацией (.digestAuth()), но она тоже не идеальна.


## 5. Как добавить секьюрность к контроллеру? (2 способа) [вопросы](#вопросы)
✅ Способ 1: Глобальная настройка через SecurityFilterChain
Определяем правила доступа к URL в конфигурации безопасности.

java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(authz -> authz
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN")
                .requestMatchers("/", "/home").permitAll()
                .anyRequest().authenticated()
            )
            .formLogin(Customizer.withDefaults());
        return http.build();
    }
}
✅ Способ 2: Аннотации на методах контроллера
Включаем поддержку аннотаций @PreAuthorize, @Secured и др.

Сначала добавляем @EnableMethodSecurity в конфигурацию:

java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class SecurityConfig { ... }
Затем в контроллере:

java
@RestController
@RequestMapping("/api")
public class MyController {

    @PreAuthorize("hasRole('ADMIN')")
    @GetMapping("/admin/data")
    public String adminData() {
        return "Sensitive data";
    }

    @PreAuthorize("hasAnyRole('USER', 'ADMIN')")
    @GetMapping("/user/data")
    public String userData() {
        return "User data";
    }
}
@PreAuthorize проверяет условие до вызова метода.

@PostAuthorize — после вызова (позволяет проверять возвращаемое значение).

@Secured — более старая аннотация (работает с ролями).

## 6. Связи таблиц: @ManyToMany и @OneToOne [вопросы](#вопросы)
В JPA (Hibernate) связи между сущностями описываются аннотациями. Понимание типов связей и их настроек критично для проектирования БД.

✅ @OneToOne (один к одному)
Используется, когда одна запись в таблице A связана ровно с одной записью в таблице B.

Варианты маппинга:

С общей внешней ключом (владеющая сторона):

java
@Entity
public class User {
    @OneToOne
    @JoinColumn(name = "profile_id")
    private Profile profile;
}
С обратной ссылкой (mappedBy):

java
@Entity
public class Profile {
    @OneToOne(mappedBy = "profile")
    private User user;
}
Можно также использовать общий первичный ключ (shared primary key), тогда @Id и @OneToOne с @MapsId.

FetchType: По умолчанию EAGER для @OneToOne. Но это может привести к излишним запросам. Лучно явно указывать LAZY и использовать @EntityGraph при необходимости.

✅ @ManyToOne и @OneToMany
@ManyToOne — многие к одному (например, много заказов принадлежат одному пользователю). Владеющая сторона (содержит @JoinColumn).

@OneToMany — обратная сторона. Обычно используется с mappedBy.

java
@Entity
public class Order {
    @ManyToOne
    @JoinColumn(name = "user_id")
    private User user;
}

@Entity
public class User {
    @OneToMany(mappedBy = "user")
    private List<Order> orders;
}
✅ @ManyToMany (многие ко многим)
Требует промежуточной таблицы.

java
@Entity
public class Student {
    @ManyToMany
    @JoinTable(
        name = "student_course",
        joinColumns = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private Set<Course> courses = new HashSet<>();
}

@Entity
public class Course {
    @ManyToMany(mappedBy = "courses")
    private Set<Student> students = new HashSet<>();
}
Проблема: нельзя добавить дополнительные поля (например, дата записи). Решение — создать промежуточную сущность (например, Enrollment) и использовать две связи @OneToMany и @ManyToOne.

🔎 Стратегии загрузки (Fetch)
LAZY — загружается только при обращении (по умолчанию для коллекций).

EAGER — загружается сразу (может привести к N+1 запросу или гигантским JOIN-ам). Лучше использовать @EntityGraph или JPQL с JOIN FETCH.

## 7. Каскады в JPA: как работают и какие бывают [вопросы](#вопросы)
Каскадирование (Cascading) — это механизм JPA, который автоматически применяет операции, выполняемые над родительской сущностью, к связанным дочерним сущностям.
Каскады (CascadeType) в JPA определяют, какие операции над родительской сущностью будут автоматически применяться к дочерним. Это уровень Hibernate, а не базы данных (хотя можно комбинировать с DDL-каскадами через @OnDelete).

✅ Типы CascadeType (из javax.persistence.CascadeType или jakarta.persistence.CascadeType)
Тип	Операция	Описание
PERSIST	EntityManager.persist()	При сохранении родителя сохраняются и дети.
MERGE	EntityManager.merge()	При слиянии родителя (обновлении) сливаются и дети.
REMOVE	EntityManager.remove()	При удалении родителя удаляются и дети.
REFRESH	EntityManager.refresh()	При обновлении состояния родителя обновляются и дети.
DETACH	EntityManager.detach()	При отсоединении родителя дети тоже отсоединяются.
ALL	Все вышеперечисленные	Включает все операции.
🛠 Пример использования
java
@Entity
public class Parent {
    @OneToMany(mappedBy = "parent", cascade = CascadeType.ALL)
    private List<Child> children = new ArrayList<>();
}

@Entity
public class Child {
    @ManyToOne
    @JoinColumn(name = "parent_id")
    private Parent parent;
}
Теперь, если вызвать entityManager.persist(parent), все дочерние объекты в списке children также будут сохранены.

⚠️ Подводные камни
Каскады не работают автоматически при удалении через JPQL (например, DELETE FROM Parent). Это операция на уровне БД, минуя контекст Hibernate.

CascadeType.REMOVE удалит детей, но если у них есть внешний ключ с NOT NULL, то перед удалением нужно обнулить ссылки на родителя. Иначе получите ошибку внешнего ключа. Решение — либо каскад на уровне БД, либо orphanRemoval = true.

orphanRemoval = true — специальная настройка для @OneToMany: удаляет дочерние записи, которые были удалены из коллекции. Она работает как каскад REMOVE, но только для «осиротевших» детей.

🔗 Каскады на уровне базы данных
Чтобы DDL содержал ON DELETE CASCADE, можно добавить @OnDelete(action = OnDeleteAction.CASCADE) к связи. Это повлияет на генерацию схемы, но не на поведение Hibernate в рантайме.

java
@ManyToOne
@OnDelete(action = OnDeleteAction.CASCADE)
private Parent parent;

## 8. Bootstrap для фронтенда [вопросы](#вопросы)
Bootstrap — это самый популярный CSS-фреймворк с открытым исходным кодом, разработанный Twitter. Он предоставляет готовые компоненты и систему сеток для быстрой разработки адаптивных и мобильных веб-интерфейсов.

🔎 Версии
Bootstrap 4 — использует Flexbox, Sass, поставляется с jQuery для некоторых компонентов.

Bootstrap 5 — отказался от jQuery, добавил новые компоненты (аккордеон, оффканвас), улучшенную сетку, утилиты.

✅ Основные возможности
Система сеток (Grid) — основана на 12 колонках. Классы .container, .row, .col-*-* (где * — размер экрана: xs, sm, md, lg, xl, xxl). Позволяет создавать адаптивные макеты.

html
<div class="container">
    <div class="row">
        <div class="col-md-8">Основной контент</div>
        <div class="col-md-4">Боковая панель</div>
    </div>
</div>
Компоненты — кнопки, навигационные панели, карточки, модальные окна, выпадающие списки, формы, таблицы, предупреждения, карусели и т.д.

Утилиты — классы для быстрого управления отступами (m-2, p-3), цветами (text-primary, bg-success), отображением (d-flex, d-none), flex-контейнерами и пр.

Адаптивность — через медиа-запросы и классы, управляющие поведением на разных разрешениях.

🛠 Примеры использования
html
<!-- Кнопка с Bootstrap -->
<button type="button" class="btn btn-primary">Primary</button>

<!-- Карточка -->
<div class="card" style="width: 18rem;">
  <img src="..." class="card-img-top" alt="...">
  <div class="card-body">
    <h5 class="card-title">Заголовок</h5>
    <p class="card-text">Текст карточки.</p>
    <a href="#" class="btn btn-primary">Перейти</a>
  </div>
</div>
🔌 Подключение
Через CDN (рекомендуется для быстрого старта):

html
<link href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/css/bootstrap.min.css" rel="stylesheet">
<script src="https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/js/bootstrap.bundle.min.js"></script>
Или через npm/yarn для сборки с кастомизацией.



## 9. REST-сервисы: преимущества и недостатки [вопросы](#вопросы)
REST (REpresentational State Transfer) — архитектурный стиль, предложенный Роем Филдингом, для построения распределённых систем. RESTful-сервисы используют HTTP как протокол передачи и оперируют ресурсами.

✅ Преимущества REST
Простота и понятность — использует стандартные HTTP-методы (GET, POST, PUT, DELETE) и статусы. Любой разработчик, знакомый с HTTP, может понять API.

Независимость от языка и платформы — клиентом может быть браузер, мобильное приложение, другой сервис на любом языке.

Масштабируемость — сервер не хранит состояние клиента (stateless), что упрощает горизонтальное масштабирование.

Кэширование — поддержка HTTP-кэширования через заголовки (Cache-Control, ETag) снижает нагрузку на сервер.

Поддержка различных форматов — JSON, XML, YAML, HTML и т.д. через согласование содержимого (content negotiation).

Использование гипермедиа (HATEOAS) — возможность динамически предоставлять ссылки на связанные ресурсы, делая API самоописываемым.

❌ Недостатки REST
Отсутствие строгого контракта — в отличие от SOAP с WSDL, REST не имеет встроенной схемы для описания API. Это может привести к несовместимости при изменениях. Проблема частично решается OpenAPI (Swagger).

Избыточность или недостаточность данных — сервер часто возвращает все поля ресурса, даже если клиенту нужно только несколько (over-fetching). Или наоборот, приходится делать несколько запросов для получения всех нужных данных (under-fetching). Альтернатива — GraphQL.

Нет встроенной безопасности — REST не определяет механизмов аутентификации/авторизации; их нужно реализовывать самостоятельно (JWT, OAuth2, Basic Auth).

Сложность с транзакциями — в распределённой среде сложно реализовать ACID-транзакции; приходится применять паттерны саги или компенсирующие транзакции.

Документирование — без OpenAPI сложно поддерживать актуальную документацию.

🔎 Принципы REST (по Филдингу)
Клиент-сервер (разделение интерфейса и хранения данных).

Отсутствие состояния (stateless).

Кэширование.

Единообразие интерфейса (uniform interface): идентификация ресурсов, манипуляция ресурсами через представления, самоописываемые сообщения, HATEOAS.

Слои (layered system).

Код по требованию (code on demand, опционально).

## 10. Форматы данных в REST [вопросы](#вопросы)
Основные форматы обмена данными в REST:

JSON (JavaScript Object Notation) — самый популярный, лёгкий, удобочитаемый. Поддерживается всеми языками.

XML (eXtensible Markup Language) — более тяжёлый, используется в legacy-системах или там, где нужна строгая валидация по схеме.

YAML — человеко-читаемый, часто для конфигураций, реже для передачи данных.

Бинарные форматы — Protocol Buffers, MessagePack — для высоконагруженных систем, где важна скорость и компактность.

Spring умеет автоматически конвертировать Java-объекты в JSON/XML с помощью HttpMessageConverter (например, Jackson для JSON).

REST-сервисы могут использовать любой формат данных, но наиболее распространены следующие:

Формат	Content-Type	Особенности
JSON	application/json	Легкий, человекочитаемый, нативная поддержка в JavaScript. Де-факто стандарт для REST API. Поддерживает вложенные структуры.
XML	application/xml	Более строгий, поддерживает схемы (XSD), используется в корпоративных системах, legacy-проектах. Может быть громоздким.
YAML	application/yaml или text/yaml	Человекочитаемый, часто используется в конфигурационных файлах, но реже для API.
HTML	text/html	Используется для веб-страниц, но может быть и форматом ответа для REST (например, для рендеринга на сервере).
Plain Text	text/plain	Простой текст, подходит для простых ответов.
Protobuf (Protocol Buffers)	application/x-protobuf	Бинарный формат от Google, компактный и быстрый, требует определения схемы (.proto). Используется в высоконагруженных системах.
MessagePack	application/msgpack	Бинарный аналог JSON, компактнее.
CBOR	application/cbor	Ещё один бинарный формат, стандарт RFC 8949.
🔎 Content Negotiation (согласование содержимого)
Клиент может указать предпочитаемый формат через заголовок Accept. Сервер выбирает подходящий HttpMessageConverter. В Spring это настраивается автоматически.

Пример запроса с указанием JSON:

text
GET /api/users/1 HTTP/1.1
Accept: application/json
Сервер может также поддерживать формат через расширение в URL: /api/users/1.json.

Настройка в Spring:

java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void configureContentNegotiation(ContentNegotiationConfigurer configurer) {
        configurer
            .favorParameter(true)
            .parameterName("mediaType")
            .defaultContentType(MediaType.APPLICATION_JSON)
            .mediaType("json", MediaType.APPLICATION_JSON)
            .mediaType("xml", MediaType.APPLICATION_XML);
    }
}


## 11. @ResponseBody, @RequestBody, ResponseEntity [вопросы](#вопросы)
Эти аннотации и класс используются для работы с телами HTTP-запросов и ответов.

Эти три элемента используются в Spring MVC для обработки тела HTTP-запроса и ответа.

✅ @RequestBody
Аннотация параметра метода контроллера, которая связывает тело HTTP-запроса с объектом Java. Spring автоматически десериализует входящие данные (например, JSON) в указанный тип с помощью подходящего HttpMessageConverter.

java
@PostMapping("/users")
public User createUser(@RequestBody User user) {
    return userService.save(user);
}
Если запрос содержит JSON {"name":"John","age":30}, то Spring создаст объект User с соответствующими полями.

Важно: метод должен быть помечен как принимающий POST, PUT или PATCH, так как GET не имеет тела.

Обработка ошибок: если тело не может быть десериализовано (например, неверный JSON), выбрасывается исключение HttpMessageNotReadableException, которое можно обработать глобально.

✅ @ResponseBody
Аннотация метода или класса, указывающая, что возвращаемое значение метода должно быть записано непосредственно в тело HTTP-ответа (а не интерпретироваться как имя представления). В сочетании с @Controller даёт тот же эффект, что и @RestController.

java
@Controller
public class MyController {
    @GetMapping("/data")
    @ResponseBody
    public String getData() {
        return "raw data";
    }
}
Можно также применять на уровне класса, тогда все методы наследуют это поведение.

✅ ResponseEntity<T>
Это полноценный объект, представляющий весь HTTP-ответ: статус, заголовки и тело. Позволяет тонко управлять ответом.

java
@PostMapping("/users")
public ResponseEntity<User> createUser(@RequestBody User user) {
    User saved = userService.save(user);
    URI location = URI.create("/users/" + saved.getId());
    return ResponseEntity
            .created(location)               // статус 201 Created + заголовок Location
            .header("Custom-Header", "value")
            .body(saved);
}
Варианты создания ResponseEntity:

Статические методы: ResponseEntity.ok(body), ResponseEntity.status(HttpStatus.CREATED).body(body), ResponseEntity.noContent().build().

Конструктор: new ResponseEntity<>(body, headers, status).

Использование с параметризацией: ResponseEntity<User> или ResponseEntity<List<User>>.

🔎 Связь с валидацией
Часто @RequestBody комбинируют с @Valid для автоматической валидации входящих данных (если есть аннотации Bean Validation). Если валидация не проходит, выбрасывается MethodArgumentNotValidException.

java
@PostMapping("/users")
public ResponseEntity<User> createUser(@Valid @RequestBody User user) { ... }

## 12. AJAX и Fetch API [вопросы](#вопросы)
🔎 AJAX (Asynchronous JavaScript and XML)
AJAX — это подход к построению веб-приложений, при котором обмен данными с сервером происходит асинхронно в фоновом режиме, без полной перезагрузки страницы. Термин был введён в 2005 году, но технология существует давно.

Основные характеристики:

Использует объект XMLHttpRequest (XHR) для отправки запросов.

Может работать с различными форматами (XML, JSON, HTML, текст).

Позволяет обновлять части страницы динамически.

Пример с XMLHttpRequest:

javascript
const xhr = new XMLHttpRequest();
xhr.open('GET', '/api/users');
xhr.onload = function() {
    if (xhr.status === 200) {
        const users = JSON.parse(xhr.responseText);
        console.log(users);
    }
};
xhr.send();
✅ Fetch API
Современная замена XMLHttpRequest, встроенная в браузеры. Основана на промисах, что делает код чище и удобнее.

Базовый пример:

javascript
fetch('/api/users')
    .then(response => {
        if (!response.ok) {
            throw new Error('Network response was not ok');
        }
        return response.json();
    })
    .then(data => console.log(data))
    .catch(error => console.error('Fetch error:', error));
Отправка POST-запроса с JSON:

javascript
fetch('/api/users', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json',
    },
    body: JSON.stringify({ name: 'John', age: 30 }),
})
.then(response => response.json())
.then(data => console.log(data));
🔎 Отличия Fetch от XHR
Промисы — Fetch возвращает Promise, что упрощает цепочки и обработку ошибок по сравнению с колбэками XHR.

Нет автопреобразования — Fetch не преобразует ответ автоматически; нужно вызывать метод .json(), .text() и т.д.

Обработка ошибок — Fetch считает успешным любой ответ, который удалось получить (даже с HTTP-статусом 404 или 500). Промис отклоняется только при сетевой ошибке (нет соединения). Поэтому нужно вручную проверять response.ok.

Куки — Fetch не отправляет куки по умолчанию. Для этого нужно указать credentials: 'include'.

Нет поддержки прогресса — Fetch не предоставляет встроенного способа отслеживать прогресс загрузки/выгрузки (можно использовать XHR или дополнительные библиотеки).

🛠 Современные альтернативы
Axios — популярная библиотека, оборачивающая XHR и предоставляющая удобный API, поддержку отмены запросов, интерцепторы.

jQuery.ajax — устаревший, но всё ещё используется в легаси-проектах.

## 13. @RestController vs @Controller [вопросы](#вопросы)
Обе аннотации используются для определения контроллеров в Spring MVC, но имеют разное назначение и поведение.

Критерий	@Controller	@RestController
Назначение	Контроллер, возвращающий представления (HTML, JSP, Thymeleaf) в классических веб-приложениях.	Специализированный контроллер для REST API, возвращающий данные (JSON/XML) напрямую в теле ответа.
Возвращаемое значение	Интерпретируется как имя представления (view), если метод не помечен @ResponseBody.	Все возвращаемые значения автоматически сериализуются в тело ответа (неявно действует @ResponseBody).
Аннотация	Является стереотипной (наследует @Component).	Является комбинацией @Controller и @ResponseBody.
Типичное использование	Веб-страницы, формы, серверный рендеринг.	JSON/XML API, микросервисы.
Обработка исключений	Можно использовать @ExceptionHandler внутри контроллера для возврата кастомной страницы ошибки.	Обычно используется @RestControllerAdvice для возврата JSON-структуры ошибки.
🛠 Примеры
@Controller с представлением:

java
@Controller
public class ViewController {
    @GetMapping("/home")
    public String home(Model model) {
        model.addAttribute("message", "Hello");
        return "home"; // ищет шаблон home.html или home.jsp
    }
}
@Controller с @ResponseBody для данных:

java
@Controller
public class DataController {
    @GetMapping("/api/data")
    @ResponseBody
    public Data getData() {
        return new Data("value");
    }
}
@RestController эквивалент:

java
@RestController
public class ApiController {
    @GetMapping("/api/data")
    public Data getData() {
        return new Data("value");
    }
}
🔎 Обработка исключений
В @Controller можно использовать @ExceptionHandler для возврата представления ошибки:

java
@ExceptionHandler(Exception.class)
public String handleError() {
    return "error";
}
В @RestController обычно создают отдельный класс с @RestControllerAdvice:

java
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(ResourceNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(ResourceNotFoundException ex) {
        return new ErrorResponse(ex.getMessage());
    }
}


## 14. RestTemplate и его методы [вопросы](#вопросы)
RestTemplate — синхронный HTTP-клиент в Spring, предназначенный для вызова REST-сервисов. Начиная с Spring 5, он помечен как устаревший (в пользу реактивного WebClient), но всё ещё широко используется в существующих проектах.

✅ Основные методы (группировка по операциям)
GET-запросы:

getForObject(url, Class<T> responseType, Object... uriVariables) — выполняет GET и возвращает тело ответа, преобразованное в объект указанного класса.

java
User user = restTemplate.getForObject("/users/{id}", User.class, 1);
getForEntity(url, Class<T> responseType, Object... uriVariables) — возвращает ResponseEntity<T>, содержащий статус, заголовки и тело.

java
ResponseEntity<User> response = restTemplate.getForEntity("/users/{id}", User.class, 1);
if (response.getStatusCode() == HttpStatus.OK) {
    User user = response.getBody();
}
POST-запросы:

postForObject(url, Object request, Class<T> responseType, Object... uriVariables) — отправляет POST с телом запроса и возвращает тело ответа.

postForEntity(...) — аналогично, но возвращает ResponseEntity.

postForLocation(url, Object request, Object... uriVariables) — отправляет POST и возвращает URL созданного ресурса (из заголовка Location).

PUT и DELETE:

put(url, Object request, Object... uriVariables) — отправляет PUT (тело запроса).

delete(url, Object... uriVariables) — отправляет DELETE.

Универсальный метод:

exchange(String url, HttpMethod method, HttpEntity<?> requestEntity, Class<T> responseType, Object... uriVariables) — позволяет выполнить любой HTTP-метод, передать заголовки, тело и получить ResponseEntity.

java
HttpHeaders headers = new HttpHeaders();
headers.setContentType(MediaType.APPLICATION_JSON);
HttpEntity<User> requestEntity = new HttpEntity<>(user, headers);
ResponseEntity<User> response = restTemplate.exchange(
    "/users", HttpMethod.POST, requestEntity, User.class);
Низкоуровневый:

execute(...) — даёт полный контроль над процессом, позволяет использовать обратные вызовы для чтения ответа.

🔎 Настройка RestTemplate
Можно настроить таймауты, перехватчики, фабрику запросов:

java
@Bean
public RestTemplate restTemplate() {
    return new RestTemplateBuilder()
            .setConnectTimeout(Duration.ofSeconds(5))
            .setReadTimeout(Duration.ofSeconds(5))
            .additionalInterceptors(new LoggingInterceptor())
            .build();
}
⚠️ Обработка ошибок
По умолчанию RestTemplate выбрасывает исключения для статусов 4xx и 5xx (например, HttpClientErrorException, HttpServerErrorException). Можно переопределить поведение, установив кастомный ResponseErrorHandler.

java
restTemplate.setErrorHandler(new DefaultResponseErrorHandler() {
    @Override
    public void handleError(ClientHttpResponse response) throws IOException {
        if (response.getStatusCode() == HttpStatus.NOT_FOUND) {
            // логика, не бросаем исключение
        } else {
            super.handleError(response);
        }
    }
});
🔄 Альтернативы
WebClient (реактивный, неблокирующий) — рекомендуется для новых проектов.

Apache HttpClient или OkHttp — можно использовать напрямую.

## 15. HTTP протокол: основы [вопросы](#вопросы)
HTTP (HyperText Transfer Protocol) — протокол прикладного уровня, лежащий в основе веба. Определяет формат сообщений и правила обмена между клиентом и сервером.

✅ Структура HTTP-запроса
text
<Метод> <URI> <Версия>
<Заголовки>

<Тело сообщения> (опционально)
Пример:

text
GET /api/users/1 HTTP/1.1
Host: example.com
Accept: application/json
✅ Структура HTTP-ответа
text
<Версия> <Код состояния> <Пояснение>
<Заголовки>

<Тело сообщения>
Пример:

text
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 45

{"id":1,"name":"John"}
🔎 Методы HTTP (глаголы)
Метод	Описание	Безопасный	Идемпотентный
GET	Получение ресурса	Да	Да
HEAD	Как GET, но без тела	Да	Да
POST	Создание ресурса или выполнение действия	Нет	Нет
PUT	Полная замена ресурса	Нет	Да
PATCH	Частичное обновление ресурса	Нет	Нет
DELETE	Удаление ресурса	Нет	Да
OPTIONS	Запрос поддерживаемых методов	Да	Да
Пояснения:

Безопасный (safe) — метод не должен изменять состояние сервера.

Идемпотентный (idempotent) — повторный запрос с теми же параметрами приводит к тому же состоянию сервера (не вызывает дополнительных изменений).

🔎 Коды состояния (статусы)
Диапазон	Категория	Примеры
1xx	Информационные	100 Continue, 101 Switching Protocols
2xx	Успех	200 OK, 201 Created, 204 No Content
3xx	Перенаправление	301 Moved Permanently, 302 Found, 304 Not Modified
4xx	Ошибка клиента	400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 405 Method Not Allowed, 429 Too Many Requests
5xx	Ошибка сервера	500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable
🔎 Основные заголовки
Общие:

Host — обязателен в HTTP/1.1.

Connection — управление соединением (Keep-Alive, close).

Запрос:

Accept — предпочитаемые форматы ответа.

Authorization — учётные данные (Basic, Bearer).

User-Agent — информация о клиенте.

Content-Type — тип тела запроса.

Content-Length — размер тела.

Ответ:

Content-Type — тип тела ответа.

Content-Length — размер тела.

Location — URL созданного ресурса (для 201 Created или редиректов).

Cache-Control — политика кэширования.

Set-Cookie — установка куки.

🔎 Кэширование
Заголовки: Cache-Control, Expires, ETag, Last-Modified.

Механизм валидации: клиент отправляет If-None-Match с ETag или If-Modified-Since, сервер отвечает 304 Not Modified, если ресурс не изменился.

🔎 Безопасность
HTTPS (HTTP over TLS) — шифрование передаваемых данных.

CORS — механизм, разрешающий запросы с других доменов.

Защита от CSRF — использование анти-CSRF токенов для запросов, изменяющих состояние.

📌 Пример полного цикла
Клиент отправляет GET-запрос на https://api.example.com/users/1 с заголовком Accept: application/json.

Сервер аутентифицирует через заголовок Authorization: Bearer <token>.

Сервер возвращает 200 OK с JSON-представлением пользователя.

Клиент отправляет PUT-запрос на тот же URL с новыми данными.

Сервер обновляет ресурс и возвращает 200 OK или 204 No Content.


Этот ответ сгенерирован AI, только для справки.
