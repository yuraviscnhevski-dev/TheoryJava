



1. Spring Security. вопросы
Spring Security — это мощный и настраиваемый фреймворк аутентификации и контроля доступа. Фактически, это стандарт для обеспечения безопасности в Spring-приложениях.

🔎 Основные возможности
Аутентификация — проверка "кто ты" (логин/пароль, токен и т.д.).

Авторизация — проверка "что тебе можно делать" (доступ к URL, методам, объектам).

Защита от уязвимостей — защита от CSRF, фиксации сессии, кликджекинга и других атак.

Интеграция — легко встраивается в веб-приложения, REST API и reactive-приложения.

Spring Security построен на цепочке Servlet-фильтров, которые перехватывают запрос и применяют правила безопасности.

2. Что такое аутентификация и авторизация? вопросы
Это два фундаментальных процесса безопасности, которые часто путают.

Аутентификация (Authentication) — это процесс проверки подлинности пользователя, доказательство, что он тот, за кого себя выдает.

Вопрос: "Кто ты?"

Пример: Ввод логина и пароля, вход по отпечатку пальца, использование токена.

Авторизация (Authorization) — это процесс определения, имеет ли аутентифицированный пользователь право на выполнение конкретного действия или доступ к ресурсу.

Вопрос: "Что тебе разрешено делать?"

Пример: После входа (аутентификации) пользователь пытается зайти в админ-панель. Система проверяет его роль (авторизация). Если роль — ADMIN, доступ разрешен, если USER — запрещен.

Веб-приложение: Сначала ты аутентифицируешься (вводишь пароль), потом для каждой страницы происходит авторизация (есть ли у тебя права).

3. Объекты Principal, Authorities, Authentication. вопросы
В Spring Security эти интерфейсы играют ключевую роль.

Principal — представляет собой "основной субъект", т.е. текущего аутентифицированного пользователя. Обычно это объект, содержащий информацию о пользователе (например, имя, ID). В Java есть базовый интерфейс java.security.Principal.

GrantedAuthority (или просто Authorities) — представляет собой "право" или "разрешение", выданное пользователю. Это могут быть роли (ROLE_ADMIN, ROLE_USER) или конкретные права (READ_AUTHORITY, WRITE_PRIVILEGE).

Authentication — это центральный интерфейс, который содержит в себе всю информацию о запросе на аутентификацию или об уже аутентифицированном пользователе. Он объединяет:

principal — пользователя.

credentials — учётные данные (например, пароль), которые обычно очищаются после успешного входа.

authorities — коллекцию прав GrantedAuthority.

Флаг isAuthenticated() — успешна ли аутентификация.

📌 Как это работает вместе:

Пользователь отправляет запрос с логином и паролем.

Spring Security создает объект Authentication (например, UsernamePasswordAuthenticationToken), который еще не аутентифицирован ( isAuthenticated() = false).

После успешной проверки данных, создается новый объект Authentication, который уже помечен как authenticated. В его principal обычно кладется объект пользователя (например, ваша кастомная реализация UserDetails), а в authorities — его права.

Этот объект сохраняется в SecurityContextHolder.getContext().setAuthentication(...), и к нему можно обратиться из любого места приложения, чтобы узнать, кто сейчас работает.

4. Чем отличается InMemoryAuthentication от basicAuthentication? вопросы
Это сравнение некорректно, так как это понятия из разных категорий. Давайте разберем их правильно.

InMemoryAuthentication — это способ хранения пользователей. Он используется для быстрой настройки, когда данные о пользователях (логины, пароли, роли) хранятся прямо в памяти приложения. Идеально для тестов и простых приложений без базы данных.

HTTP Basic Authentication — это механизм (способ) аутентификации, определенный в протоколе HTTP. Браузер или клиент отправляет логин и пароль в заголовке Authorization, закодированные в Base64. Это не про хранение, а про передачу данных.

✅ Их можно и нужно комбинировать:

java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(authz -> authz
            .anyRequest().authenticated()
        )
        .httpBasic(Customizer.withDefaults()); // Включаем HTTP Basic
    return http.build();
}

@Bean
public UserDetailsService users() {
    UserDetails user = User.builder()
        .username("user")
        .password("{noop}password") // {noop} означает, что пароль хранится в открытом виде
        .roles("USER")
        .build();
    return new InMemoryUserDetailsManager(user); // Храним в памяти
}
В этом примере мы используем InMemoryAuthentication для хранения пользователя "user", и просим Spring Security использовать HTTP Basic для получения логина и пароля от клиента.

5. Как мы можем добавить секьюрность к контроллеру? (минимум 2 способа). вопросы
Существует несколько способов ограничить доступ к методам контроллера.

1️⃣ Глобальная настройка через SecurityFilterChain (на уровне URL)

Этот способ определяет правила доступа к определенным путям (endpoints) для всего приложения. Это наиболее декларативный способ.

java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(authz -> authz
                .requestMatchers("/admin/**").hasRole("ADMIN")       // Только для ADMIN
                .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN") // Для USER и ADMIN
                .requestMatchers("/", "/home", "/public/**").permitAll()  // Для всех
                .anyRequest().authenticated()                       // Все остальные запросы требуют аутентификации
            )
            .formLogin(Customizer.withDefaults()); // Включаем стандартную форму логина
        return http.build();
    }

    @Bean
    public UserDetailsService users() {
        // ... настройка пользователей
    }
}
2️⃣ Через аннотации на методах контроллера (на уровне метода)

Этот способ более точечный. Мы включаем глобальную поддержку аннотаций, а затем вешаем их прямо на методы контроллера.

Сначала нужно включить обработку аннотаций, добавив @EnableMethodSecurity в конфигурацию:

java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity // Включаем поддержку аннотаций
public class SecurityConfig {
    // ... остальная конфигурация
}
Теперь в контроллере можно использовать аннотации:

java
@RestController
@RequestMapping("/api/data")
public class DataController {

    @GetMapping("/public")
    public String publicData() {
        return "Это публичные данные";
    }

    @PreAuthorize("hasRole('USER')") // Только для пользователей с ролью USER
    @GetMapping("/user")
    public String userData() {
        return "Это данные для пользователей";
    }

    @PreAuthorize("hasRole('ADMIN')") // Только для ADMIN
    @DeleteMapping("/admin/{id}")
    public String deleteData(@PathVariable Long id) {
        return "Данные с ID " + id + " удалены";
    }

    @PostAuthorize("returnObject.owner == authentication.name") // Проверка после выполнения метода
    @GetMapping("/secure/{id}")
    public Data getSecureData(@PathVariable Long id) {
        // ... логика получения Data
    }
}
@PreAuthorize — проверяет условие до вызова метода.

@PostAuthorize — проверяет условие после вызова метода (часто используется для проверки прав на возвращаемый объект).

@Secured — более старая аннотация, работает с ролями (например, @Secured("ROLE_ADMIN")).

@RolesAllowed — стандартная аннотация Java (JSR-250), аналогична @Secured.

6. Связи таблиц many-to-many one-to-one вопросы
В ORM (например, JPA/Hibernate) связи между сущностями (таблицами) описываются с помощью аннотаций.

1️⃣ One-to-One (Один к одному)

Каждая запись в одной таблице соответствует ровно одной записи в другой.
Пример: Пользователь (User) и его профиль (Profile).

java
@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String username;

    // Указываем, что связь находится на стороне Profile (mappedBy)
    @OneToOne(mappedBy = "user", cascade = CascadeType.ALL)
    private Profile profile;
    // ... геттеры/сеттеры
}

@Entity
public class Profile {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String firstName;
    private String lastName;

    // Владеющая сторона связи. Здесь хранится внешний ключ (user_id)
    @OneToOne
    @JoinColumn(name = "user_id") // Имя столбца для внешнего ключа
    private User user;
    // ... геттеры/сеттеры
}
2️⃣ Many-to-Many (Многие ко многим)

Множество записей из одной таблицы связано со множеством записей из другой. Реализуется через третью (промежуточную) таблицу.
Пример: Студенты (Student) и Курсы (Course). Один студент может посещать много курсов, на один курс может ходить много студентов.

java
@Entity
public class Student {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    @ManyToMany
    @JoinTable(
        name = "student_course", // Имя промежуточной таблицы
        joinColumns = @JoinColumn(name = "student_id"), // Колонка для Student
        inverseJoinColumns = @JoinColumn(name = "course_id") // Колонка для Course
    )
    private Set<Course> courses = new HashSet<>();
    // ... геттеры/сеттеры
}

@Entity
public class Course {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;

    // Обратная сторона связи (mappedBy указывает на поле в Student)
    @ManyToMany(mappedBy = "courses")
    private Set<Student> students = new HashSet<>();
    // ... геттеры/сеттеры
}
7. Как работают каскады для таблиц и какие они бывают. вопросы
Каскадирование (Cascading) — это механизм JPA, который позволяет автоматически применять операции, выполняемые над родительской сущностью, к связанным с ней дочерним сущностям.

🔎 Типы каскадов (CascadeType)
Тип	Описание
ALL	Применяются все операции (PERSIST, MERGE, REMOVE, REFRESH, DETACH).
PERSIST	При сохранении родителя сохраняются и его дети.
MERGE	При обновлении родителя обновляются и его дети.
REMOVE	При удалении родителя удаляются и его дети.
REFRESH	При обновлении состояния родителя из БД обновляются и дети.
DETACH	При отсоединении родителя от контекста отсоединяются и дети.
📌 Как это работает:

Допустим, у нас есть пользователь и его заказы (связь @OneToMany).

java
@Entity
public class User {
    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL) // Указываем каскад ALL
    private List<Order> orders = new ArrayList<>();
    // ...
}
Теперь:

Сохранение: Если мы сохраняем нового User с новыми Order в списке, вызов entityManager.persist(user) автоматически сохранит и все Order.

Удаление: Если мы удаляем User (entityManager.remove(user)), то все его Order также будут удалены из БД.

⚠️ Важно: Каскадирование работает на уровне JPA (в рамках одной транзакции и контекста持久化). Оно не устанавливает автоматически каскады ON DELETE CASCADE в самой базе данных. Для этого нужно настраивать DDL отдельно.

8. Bootstrap. вопросы
Bootstrap — это не вопрос по Spring, а популярный HTML, CSS и JS фреймворк для разработки веб-интерфейсов.

Зачем нужен: Позволяет быстро создавать адаптивные (responsive) и стильные сайты, не углубляясь в тонкости CSS.

Как используется в Spring: В Spring MVC приложениях его часто подключают для создания фронтенда. Можно добавить зависимость через WebJars, чтобы библиотека хранилась внутри проекта.

Основные компоненты:

Сетка (Grid) — 12-колоночная система для адаптивной верстки.

Готовые компоненты — кнопки, навигационные панели, карточки, модальные окна, формы и т.д.

Утилиты — классы для быстрого управления отступами, цветами, текстом.

Пример подключения через WebJars в pom.xml:

xml
<dependency>
    <groupId>org.webjars</groupId>
    <artifactId>bootstrap</artifactId>
    <version>5.3.0</version>
</dependency>
9. Rest-сервисы. Их преимущества и недостатки вопросы
REST (Representational State Transfer) — это архитектурный стиль взаимодействия компонентов распределенного приложения в сети. RESTful-сервис использует HTTP и его методы.

✅ Преимущества
Простота и понятность: Использует стандартные HTTP методы (GET, POST, PUT, DELETE) и статусы (200 OK, 404 Not Found).

Масштабируемость: Серверы не хранят состояние клиента (stateless), что упрощает горизонтальное масштабирование.

Гибкость и независимость: Клиент и сервер развиваются независимо, обмениваясь данными в стандартных форматах.

Кэширование: Ответы могут быть явно помечены как кэшируемые или нет, что снижает нагрузку.

Универсальность: Может использоваться любым клиентом, способным отправлять HTTP-запросы (браузеры, мобильные приложения, IoT-устройства).

❌ Недостатки
Отсутствие строгой спецификации: В отличие от SOAP, нет единого стандарта (WSDL), что может приводить к разной трактовке принципов.

Проблема избыточности (over-fetching): Сервер часто возвращает больше данных, чем нужно клиенту (если не использовать такие техники, как GraphQL или проекции).

Проблема множественных запросов (under-fetching): Для получения полной информации клиенту может потребоваться сделать несколько последовательных запросов.

Сложности с безопасностью: Требуется дополнительная настройка HTTPS, аутентификации (JWT, OAuth2) и авторизации.

10. Форматы данных, использующиеся в REST-сервисах вопросы
Основной и самый популярный формат — JSON (JavaScript Object Notation). Он легкий, читаемый и легко парсится в любом языке программирования.

Другие форматы:

XML (eXtensible Markup Language): Более тяжелый и строгий формат. Используется в legacy-системах или там, где важна валидация по XSD-схемам.

HTML: Иногда REST API может возвращать готовые фрагменты HTML для вставки на страницу (например, при использовании HTMX).

YAML: Человеко-читаемый формат, часто используется для конфигураций, реже для передачи данных в API.

Бинарные форматы: Protocol Buffers (protobuf) от Google или MessagePack, используются для высоконагруженных систем, где важна скорость и компактность.

Spring умеет автоматически конвертировать Java-объекты в JSON и XML благодаря HttpMessageConverter (например, Jackson для JSON, JAXB для XML).

11. Что такое responseBody, requestBody, ResponseEntity вопросы
Это ключевые элементы для работы с телами HTTP-запросов и ответов в Spring MVC.

@RequestBody — аннотация, которая связывает тело HTTP-запроса с параметром метода в контроллере. Spring автоматически десериализует JSON/XML из запроса в Java-объект.

java
@PostMapping("/users")
public User createUser(@RequestBody User newUser) { // JSON из запроса станет объектом User
    return service.save(newUser);
}
@ResponseBody — аннотация, которая указывает, что возвращаемое значение метода должно быть записано прямо в тело HTTP-ответа (в виде JSON/XML), а не интерпретироваться как имя представления (как в классическом MVC).

java
@GetMapping("/user/{id}")
@ResponseBody
public User getUser(@PathVariable Long id) {
    return service.findById(id); // Объект User станет JSON-ом в ответе
}
Важно: @RestController включает в себя @Controller и @ResponseBody на всех методах, поэтому отдельно @ResponseBody ставить не нужно.

ResponseEntity<T> — это специальный объект, который представляет весь HTTP-ответ: тело, заголовки и статус. Он дает полный контроль над тем, что вернется клиенту.

java
@PostMapping("/users")
public ResponseEntity<User> createUser(@RequestBody User user) {
    User saved = service.save(user);
    return ResponseEntity
            .status(HttpStatus.CREATED)       // Статус 201
            .header("Custom-Header", "value") // Заголовки
            .body(saved);                     // Тело ответа
}
12. Что такое AJAX/fetch? вопросы
Это технологии для асинхронного общения браузера с сервером без перезагрузки всей страницы.

AJAX (Asynchronous JavaScript And XML) — это подход (методология), а не конкретная технология. Он использует встроенный в браузер объект XMLHttpRequest для отправки HTTP-запросов и обработки ответов в фоновом режиме. Несмотря на название "XML", сейчас чаще всего используется JSON.

Fetch API — это современная, более мощная и гибкая замена XMLHttpRequest, встроенная в JavaScript. Она основана на промисах (Promises), что делает код чище и проще для понимания.

📌 Пример использования fetch в JavaScript:

javascript
// Отправляем GET-запрос
fetch('/api/users/1')
  .then(response => {
    if (!response.ok) {
      throw new Error('Network response was not ok');
    }
    return response.json(); // Парсим ответ как JSON
  })
  .then(data => {
    console.log('Данные пользователя:', data); // Работаем с данными
    document.getElementById('username').textContent = data.name;
  })
  .catch(error => {
    console.error('Ошибка:', error);
  });

// Отправляем POST-запрос с данными
const newUser = { name: 'Новый Пользователь', email: 'new@user.com' };

fetch('/api/users', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json', // Говорим, что отправляем JSON
  },
  body: JSON.stringify(newUser), // Превращаем объект в JSON-строку
})
.then(response => response.json())
.then(data => console.log('Успех:', data))
.catch(error => console.error('Ошибка:', error));
13. Чем аннотация RestController отличается от Controller вопросы
Это два основных типа контроллеров в Spring.

Характеристика	@Controller	@RestController
Основная задача	Используется в классических MVC-приложениях, где нужно возвращать представления (HTML, JSP, Thymeleaf).	Специализирован для создания RESTful API, где нужно возвращать данные (JSON, XML).
Возвращаемое значение	По умолчанию имя представления, которое будет отрендерено.	По умолчанию объект данных, который автоматически сериализуется в JSON/XML и пишется в тело ответа.
Комбинация	Это просто @Component + возможность возвращать ModelAndView.	Это @Controller + @ResponseBody на всех методах.
Пример	Возвращает "home", что приводит к отображению home.html.	Возвращает new User("John"), что приводит к ответу {"name":"John"}.
java
// JSP страница /WEB-INF/views/home.jsp
@Controller
public class ViewController {
    @GetMapping("/home")
    public String home(Model model) {
        model.addAttribute("message", "Hello World");
        return "home"; // Возвращаем имя представления
    }
}
java
// REST API
@RestController
public class ApiController {
    @GetMapping("/api/user")
    public User getUser() {
        return new User("John"); // Возвращаем объект -> станет JSON
    }
}
14. RestTemplate и его методы вопросы
RestTemplate — это синхронный HTTP-клиент, предоставляемый Spring для работы с RESTful-сервисами. Он упрощает отправку запросов и обработку ответов.

⚠️ Важно: Начиная со Spring 5, RestTemplate находится в режиме обслуживания (поддержка, но новые возможности не добавляются). Рекомендуется использовать неблокирующий WebClient. Однако RestTemplate все еще очень широко используется.

🔎 Основные методы RestTemplate
Категория	Метод	Описание
GET	getForObject(url, классОтвета, ...)	Отправляет GET-запрос и возвращает тело ответа, преобразованное в указанный класс.
getForEntity(url, классОтвета, ...)	Отправляет GET-запрос и возвращает полный ответ в виде ResponseEntity (статус, заголовки, тело).
POST	postForObject(url, запрос, классОтвета, ...)	Отправляет POST-запрос с телом запрос и возвращает тело ответа.
postForEntity(url, запрос, классОтвета, ...)	Отправляет POST-запрос и возвращает ResponseEntity.
postForLocation(url, запрос, ...)	Отправляет POST-запрос и возвращает URL созданного ресурса (из заголовка Location).
PUT	put(url, запрос, ...)	Отправляет PUT-запрос для обновления ресурса. Ничего не возвращает.
DELETE	delete(url, ...)	Отправляет DELETE-запрос.
Универсальный	exchange(url, метод, httpEntity, классОтвета, ...)	Самый гибкий метод. Позволяет задать HTTP-метод, заголовки, тело запроса и тип ответа.
execute(url, метод, requestCallback, responseExtractor, ...)	Максимальный контроль, позволяет выполнять действия на сыром уровне запроса/ответа.
📌 Пример использования:

java
@Service
public class UserServiceClient {

    private final RestTemplate restTemplate;

    public UserServiceClient(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    public User getUserById(Long id) {
        String url = "https://jsonplaceholder.typicode.com/users/{id}";
        // GET-запрос и преобразование ответа в объект User
        return restTemplate.getForObject(url, User.class, id);
    }

    public User createUser(User newUser) {
        String url = "https://jsonplaceholder.typicode.com/users";
        // POST-запрос, отправка newUser, получение созданного пользователя
        return restTemplate.postForObject(url, newUser, User.class);
    }

    public ResponseEntity<User> getUserWithHeaders(Long id) {
        String url = "https://jsonplaceholder.typicode.com/users/{id}";
        // GET-запрос с получением статуса и заголовков
        return restTemplate.getForEntity(url, User.class, id);
    }
}
15. HTTP протокол вопросы
HTTP (HyperText Transfer Protocol) — это протокол прикладного уровня, который лежит в основе обмена данными в World Wide Web. Это основа для REST.

🔎 Ключевые понятия
1. HTTP Методы (Глаголы): Указывают желаемое действие над ресурсом.

GET — получить ресурс (безопасный, идемпотентный).

POST — создать новый ресурс (не идемпотентный).

PUT — полностью заменить существующий ресурс (идемпотентный).

PATCH — частично обновить ресурс.

DELETE — удалить ресурс (идемпотентный).

HEAD — получить только заголовки, как при GET, но без тела.

OPTIONS — узнать, какие методы поддерживает сервер для ресурса.

2. HTTP Статусы (Коды ответа): Трехзначный код, сообщающий клиенту результат его запроса.

1xx (Информационные): 100 Continue.

2xx (Успех): 200 OK, 201 Created, 204 No Content.

3xx (Перенаправление): 301 Moved Permanently, 302 Found, 304 Not Modified.

4xx (Ошибка клиента): 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found.

5xx (Ошибка сервера): 500 Internal Server Error, 503 Service Unavailable.

3. HTTP Заголовки (Headers): Метаданные запроса или ответа.

Запроса (Request Headers): Host: example.com, User-Agent: ..., Accept: application/json, Authorization: Bearer <token>.

Ответа (Response Headers): Content-Type: application/json, Content-Length: 123, Cache-Control: no-cache, Set-Cookie: ....

4. Структура сообщения:

Стартовая строка: (для запроса) GET /users HTTP/1.1 или (для ответа) HTTP/1.1 200 OK.

Заголовки: пара "ключ: значение".

Пустая строка: разделяет заголовки и тело.

Тело сообщения (Body): (необязательно) передаваемые данные (HTML, JSON, изображение).
