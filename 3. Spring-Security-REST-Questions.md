## 1. Что такое Spring Security? [вопросы](#вопросы)

 Spring Security — это мощный и настраиваемый фреймворк аутентификации и контроля доступа, который является стандартом для обеспечения безопасности в Spring-приложениях.

🔎 Основные возможности
Аутентификация — проверка подлинности пользователя (логин/пароль, токены, OAuth2 и т.д.).

Авторизация — проверка прав доступа к ресурсам (по ролям,权限).

Защита от атак — защита от CSRF, фиксации сессии, clickjacking и других уязвимостей.

Интеграция — легко встраивается в веб-приложения, REST API, reactive-приложения.

Поддержка протоколов — LDAP, OAuth2, JWT, OpenID Connect и др.

Spring Security построен на цепочке Servlet-фильтров, которые перехватывают HTTP-запросы и применяют правила безопасности до того, как запрос достигнет контроллера.

## 2. Что такое аутентификация и авторизация? [вопросы](#вопросы)
Это два фундаментальных процесса безопасности, которые часто путают.

🚀 Аутентификация (Authentication)
Вопрос: "Кто ты?"

Процесс: Проверка учётных данных пользователя (логин + пароль, токен, биометрия).

Результат: Если данные верны, система запоминает, что пользователь — это именно он.

🚀 Авторизация (Authorization)
Вопрос: "Что тебе разрешено делать?"

Процесс: Проверка прав доступа после успешной аутентификации (например, имеет ли пользователь роль ADMIN для доступа к /admin).

Результат: Доступ к ресурсу либо разрешается, либо возвращается ошибка 403 Forbidden.

Пример:

Пользователь вводит логин/пароль — это аутентификация.

После входа он пытается зайти в админку — система проверяет его роль — это авторизация.

## 3. Объекты Principal, Authorities, Authentication [вопросы](#вопросы)
В Spring Security эти интерфейсы являются ключевыми.

🔑 Principal
Представляет собой текущего аутентифицированного пользователя. Обычно это объект, содержащий имя пользователя или более подробную информацию (например, экземпляр UserDetails).

Интерфейс java.security.Principal.

🔑 GrantedAuthority (Authorities)
Представляет разрешения, выданные пользователю. Могут быть ролями (ROLE_ADMIN, ROLE_USER) или конкретными правами (READ_PRIVILEGE).

Хранятся в виде списка внутри объекта Authentication.

🔑 Authentication
Основной интерфейс, который объединяет:

principal — пользователь.

credentials — учётные данные (обычно пароль, после аутентификации очищается).

authorities — права пользователя.

isAuthenticated() — флаг успешной аутентификации.

После успешного входа объект Authentication помещается в SecurityContextHolder.

Пример получения текущего пользователя:

java
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String username = auth.getName();
Collection<? extends GrantedAuthority> authorities = auth.getAuthorities();
## 4. Чем отличается InMemoryAuthentication от basicAuthentication? [вопросы](#вопросы)
Это сравнение не совсем корректно, так как это понятия из разных категорий.

InMemoryAuthentication — это способ хранения пользователей. Позволяет задать список пользователей прямо в коде (в памяти). Используется для тестов и простых приложений.

HTTP Basic Authentication — это механизм аутентификации, при котором логин и пароль передаются в заголовке Authorization в закодированном виде (Base64). Это способ передачи данных, а не их хранения.

Как они сочетаются:
Вы можете использовать InMemoryAuthentication для хранения пользователей, а httpBasic() — для их аутентификации.

Пример конфигурации:

java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(authz -> authz.anyRequest().authenticated())
        .httpBasic(Customizer.withDefaults()); // используем HTTP Basic
    return http.build();
}

@Bean
public UserDetailsService users() {
    UserDetails user = User.withUsername("user")
        .password("{noop}password") // {noop} — без шифрования
        .roles("USER")
        .build();
    return new InMemoryUserDetailsManager(user); // храним в памяти
}
## 5. Как добавить секьюрность к контроллеру? (2 способа) [вопросы](#вопросы)
✅ Способ 1: Глобальная настройка через SecurityFilterChain
Определяем правила доступа к URL в конфигурации безопасности.

java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(authz -> authz
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN")
                .requestMatchers("/", "/home").permitAll()
                .anyRequest().authenticated()
            )
            .formLogin(Customizer.withDefaults());
        return http.build();
    }
}
✅ Способ 2: Аннотации на методах контроллера
Включаем поддержку аннотаций @PreAuthorize, @Secured и др.

Сначала добавляем @EnableMethodSecurity в конфигурацию:

java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class SecurityConfig { ... }
Затем в контроллере:

java
@RestController
@RequestMapping("/api")
public class MyController {

    @PreAuthorize("hasRole('ADMIN')")
    @GetMapping("/admin/data")
    public String adminData() {
        return "Sensitive data";
    }

    @PreAuthorize("hasAnyRole('USER', 'ADMIN')")
    @GetMapping("/user/data")
    public String userData() {
        return "User data";
    }
}
@PreAuthorize проверяет условие до вызова метода.

@PostAuthorize — после вызова (позволяет проверять возвращаемое значение).

@Secured — более старая аннотация (работает с ролями).

## 6. Связи таблиц: @ManyToMany и @OneToOne [вопросы](#вопросы)
В JPA для описания связей между сущностями используются аннотации.

🟢 @OneToOne (один к одному)
Каждая запись в одной таблице соответствует ровно одной записи в другой.
Пример: User и Profile.

java
@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String username;

    @OneToOne(mappedBy = "user", cascade = CascadeType.ALL)
    private Profile profile;
    // геттеры/сеттеры
}

@Entity
public class Profile {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String firstName;
    private String lastName;

    @OneToOne
    @JoinColumn(name = "user_id") // внешний ключ
    private User user;
    // геттеры/сеттеры
}
🟢 @ManyToMany (многие ко многим)
Множество записей из одной таблицы связано со множеством записей из другой. Используется промежуточная таблица.
Пример: Student и Course.

java
@Entity
public class Student {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;

    @ManyToMany
    @JoinTable(
        name = "student_course",
        joinColumns = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private Set<Course> courses = new HashSet<>();
    // геттеры/сеттеры
}

@Entity
public class Course {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String title;

    @ManyToMany(mappedBy = "courses")
    private Set<Student> students = new HashSet<>();
    // геттеры/сеттеры
}
## 7. Каскады в JPA: как работают и какие бывают [вопросы](#вопросы)
Каскадирование (Cascading) — это механизм JPA, который автоматически применяет операции, выполняемые над родительской сущностью, к связанным дочерним сущностям.

🔎 Типы каскадов (CascadeType)
Тип	Описание
ALL	Все операции (PERSIST, MERGE, REMOVE, REFRESH, DETACH)
PERSIST	При сохранении родителя сохраняются и дети
MERGE	При обновлении родителя обновляются и дети
REMOVE	При удалении родителя удаляются и дети
REFRESH	При обновлении состояния родителя из БД обновляются и дети
DETACH	При отсоединении родителя отсоединяются и дети
Пример использования:

java
@Entity
public class User {
    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL)
    private List<Order> orders;
}
Теперь при сохранении пользователя сохранятся и все его заказы, при удалении пользователя удалятся и заказы.

Важно: Каскады работают на уровне JPA (в рамках одной транзакции). Они не создают автоматически ON DELETE CASCADE в базе данных — это нужно настраивать отдельно.

## 8. Bootstrap для фронтенда [вопросы](#вопросы)
Bootstrap — это популярный CSS-фреймворк для разработки адаптивных и стильных веб-интерфейсов. Он предоставляет готовые компоненты (кнопки, навигационные панели, формы, сетки) и утилиты.

🔧 Как использовать в Spring MVC
Подключить через WebJars (менеджер зависимостей) в pom.xml:

xml
<dependency>
    <groupId>org.webjars</groupId>
    <artifactId>bootstrap</artifactId>
    <version>5.3.0</version>
</dependency>
В шаблоне (например, Thymeleaf) подключить CSS и JS:

html
<link rel="stylesheet" th:href="@{/webjars/bootstrap/5.3.0/css/bootstrap.min.css}">
<script th:src="@{/webjars/bootstrap/5.3.0/js/bootstrap.bundle.min.js}"></script>
Использовать классы Bootstrap в HTML.

## 9. REST-сервисы: преимущества и недостатки [вопросы](#вопросы)
REST (Representational State Transfer) — архитектурный стиль построения распределённых систем, использующий HTTP.

✅ Преимущества
Простота — использует стандартные HTTP-методы и статусы.

Масштабируемость — сервер не хранит состояние клиента (stateless).

Гибкость — клиент и сервер могут развиваться независимо.

Кэширование — ответы можно кэшировать.

Универсальность — подходит для любых клиентов (браузеры, мобильные приложения).

❌ Недостатки
Отсутствие строгой спецификации — разные реализации могут трактовать принципы по-разному.

Избыточность данных — сервер часто возвращает больше данных, чем нужно (over-fetching).

Множественные запросы — для получения сложных данных может потребоваться несколько запросов (under-fetching).

Безопасность — требует дополнительной настройки (HTTPS, JWT, OAuth2).

## 10. Форматы данных в REST [вопросы](#вопросы)
Основные форматы обмена данными в REST:

JSON (JavaScript Object Notation) — самый популярный, лёгкий, удобочитаемый. Поддерживается всеми языками.

XML (eXtensible Markup Language) — более тяжёлый, используется в legacy-системах или там, где нужна строгая валидация по схеме.

YAML — человеко-читаемый, часто для конфигураций, реже для передачи данных.

Бинарные форматы — Protocol Buffers, MessagePack — для высоконагруженных систем, где важна скорость и компактность.

Spring умеет автоматически конвертировать Java-объекты в JSON/XML с помощью HttpMessageConverter (например, Jackson для JSON).

## 11. @ResponseBody, @RequestBody, ResponseEntity [вопросы](#вопросы)
Эти аннотации и класс используются для работы с телами HTTP-запросов и ответов.

📌 @RequestBody
Связывает тело HTTP-запроса с параметром метода. Spring автоматически десериализует JSON/XML в Java-объект.

java
@PostMapping("/users")
public User createUser(@RequestBody User newUser) {
    return service.save(newUser);
}
📌 @ResponseBody
Указывает, что возвращаемое значение метода должно быть записано прямо в тело HTTP-ответа (в JSON/XML), а не интерпретироваться как имя представления.

java
@GetMapping("/user/{id}")
@ResponseBody
public User getUser(@PathVariable Long id) {
    return service.findById(id);
}
Примечание: @RestController включает в себя @Controller и @ResponseBody на всех методах.

📌 ResponseEntity<T>
Представляет полный HTTP-ответ: статус, заголовки и тело. Позволяет гибко управлять ответом.

java
@PostMapping("/users")
public ResponseEntity<User> createUser(@RequestBody User user) {
    User saved = service.save(user);
    return ResponseEntity
            .status(HttpStatus.CREATED)
            .header("Custom-Header", "value")
            .body(saved);
}
## 12. AJAX и Fetch API [вопросы](#вопросы)
AJAX (Asynchronous JavaScript And XML) — подход к асинхронному общению браузера с сервером без перезагрузки страницы. Исторически использовал объект XMLHttpRequest.

Fetch API — современная замена XMLHttpRequest, встроенная в JavaScript, основанная на промисах.

Пример использования fetch:

javascript
// GET-запрос
fetch('/api/users/1')
  .then(response => response.json())
  .then(data => console.log(data))
  .catch(error => console.error('Error:', error));

// POST-запрос с JSON
const newUser = { name: 'John', email: 'john@example.com' };
fetch('/api/users', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(newUser)
})
.then(response => response.json())
.then(data => console.log('User created:', data));
## 13. @RestController vs @Controller [вопросы](#вопросы)
Характеристика	@Controller	@RestController
Назначение	Возвращает представления (HTML, JSP, Thymeleaf)	Возвращает данные (JSON, XML)
Возвращаемое значение	Имя представления или ModelAndView	Объект, сериализуемый в тело ответа
Содержит	@Component	@Controller + @ResponseBody на всех методах
Пример метода	return "home";	return new User("John");
Примеры:

java
@Controller
public class ViewController {
    @GetMapping("/home")
    public String home(Model model) {
        model.addAttribute("message", "Hello");
        return "home"; // home.html
    }
}

@RestController
public class ApiController {
    @GetMapping("/api/user")
    public User getUser() {
        return new User("John"); // JSON: {"name":"John"}
    }
}
## 14. RestTemplate и его методы [вопросы](#вопросы)
RestTemplate — синхронный HTTP-клиент Spring для вызова REST-сервисов. Начиная с Spring 5, рекомендуется использовать WebClient, но RestTemplate всё ещё широко применяется.

🔎 Основные методы
Метод	Описание
getForObject(url, class, ...)	GET, возвращает тело ответа, преобразованное в указанный класс
getForEntity(url, class, ...)	GET, возвращает ResponseEntity (статус, заголовки, тело)
postForObject(url, request, class, ...)	POST, отправляет объект и возвращает тело ответа
postForEntity(url, request, class, ...)	POST, возвращает ResponseEntity
put(url, request, ...)	PUT, обновляет ресурс
delete(url, ...)	DELETE, удаляет ресурс
exchange(url, method, httpEntity, responseType, ...)	Универсальный метод для любого HTTP-метода
Пример:

java
@Service
public class UserClient {
    private final RestTemplate restTemplate = new RestTemplate();

    public User getUser(Long id) {
        String url = "https://api.example.com/users/{id}";
        return restTemplate.getForObject(url, User.class, id);
    }

    public User createUser(User newUser) {
        String url = "https://api.example.com/users";
        return restTemplate.postForObject(url, newUser, User.class);
    }
}
## 15. HTTP протокол: основы [вопросы](#вопросы)
HTTP (HyperText Transfer Protocol) — протокол прикладного уровня, основа веб-взаимодействия. Лежит в основе REST.

🔎 Ключевые понятия
1. HTTP методы (глаголы)

GET — получить ресурс (безопасный, идемпотентный)

POST — создать новый ресурс

PUT — полностью заменить ресурс (идемпотентный)

PATCH — частично обновить ресурс

DELETE — удалить ресурс (идемпотентный)

HEAD — получить только заголовки

OPTIONS — узнать поддерживаемые методы

2. HTTP статусы (коды ответа)

1xx — информационные (100 Continue)

2xx — успех (200 OK, 201 Created, 204 No Content)

3xx — перенаправление (301 Moved Permanently, 302 Found)

4xx — ошибка клиента (400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found)

5xx — ошибка сервера (500 Internal Server Error, 503 Service Unavailable)

3. HTTP заголовки

Запроса: Host, User-Agent, Accept, Authorization, Content-Type

Ответа: Content-Type, Content-Length, Cache-Control, Set-Cookie

4. Структура сообщения

Стартовая строка (Request Line / Status Line)

Заголовки (headers)

Пустая строка

Тело сообщения (body) — опционально

Пример HTTP-запроса:

text
GET /api/users/1 HTTP/1.1
Host: example.com
Accept: application/json
Пример HTTP-ответа:

text
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 45

{"id":1,"name":"John","email":"john@example.com"}
