## <a id="q1"></a>1. Spring Security: что такое авторизация и аутентификация?

Это два фундаментальных понятия безопасности, которые часто путают:

| Термин | Определение | Пример |
|--------|-------------|--------|
| **Аутентификация** | Процесс проверки, кто ты такой. Подтверждение личности пользователя | Ввод логина и пароля, проверка по отпечатку пальца |
| **Авторизация** | Процесс проверки, что тебе разрешено делать. Определение прав доступа | Доступ к странице `/admin` только для роли ADMIN |

**Простыми словами:**
- Аутентификация отвечает на вопрос **"Кто ты?"**
- Авторизация отвечает на вопрос **"Что тебе можно?"**

В Spring Security эти процессы разделены и настраиваются отдельно. Аутентификация происходит через `AuthenticationManager`, а авторизация — через конфигурацию `HttpSecurity`.

## <a id="q2"></a>2. Объекты Principal, Authorities, Authentication

Это ключевые интерфейсы Spring Security, которые нужно понимать:

| Интерфейс | Назначение | Ключевые методы |
|-----------|------------|-----------------|
| **Principal** | Базовое представление пользователя (из Java SE) | `getName()` |
| **Authentication** | Основной объект аутентификации в Spring, расширяет Principal | `getPrincipal()`, `getCredentials()`, `getAuthorities()`, `isAuthenticated()` |
| **GrantedAuthority** | Представляет разрешение/роль пользователя | `getAuthority()` |

**Подробнее об Authentication:**

`Authentication` — это центральный интерфейс, который представляет:
- **Запрос на аутентификацию** (еще не проверенный пользователь)
- **Аутентифицированного пользователя** (уже проверенный)

Основные методы:
- `getPrincipal()` — возвращает идентификатор пользователя (обычно объект `UserDetails`)
- `getCredentials()` — возвращает учетные данные (обычно пароль, после аутентификации очищается)
- `getAuthorities()` — возвращает коллекцию разрешений пользователя
- `isAuthenticated()` — показывает, прошел ли пользователь аутентификацию

**Пример иерархии:**
SecurityContextHolder → SecurityContext → Authentication → Principal + Authorities

## <a id="q3"></a>3. Чем отличается InMemoryAuthentication от BasicAuthentication?

Это вообще разные вещи, их часто путают. Давай разберем:

| Критерий | InMemoryAuthentication | BasicAuthentication |
|----------|------------------------|---------------------|
| **Что это** | Способ **хранения** пользователей | Способ **передачи** учетных данных |
| **Суть** | Хранение учетных записей прямо в памяти приложения | HTTP-аутентификация через заголовок `Authorization` |
| **Где хранятся данные** | В памяти (оперативная память) | Не хранятся, передаются с каждым запросом |
| **Когда использовать** | Тестирование, прототипы, небольшие приложения | API, сервисы, stateless-аутентификация |

**InMemoryAuthentication — пример:**
```java
@Configuration
public class SecurityConfig {
    
    @Bean
    public UserDetailsService users() {
        UserDetails user = User.builder()
            .username("user")
            .password("{noop}password")
            .roles("USER")
            .build();
        return new InMemoryUserDetailsManager(user);
    }
}
BasicAuthentication — пример:

java
@Configuration
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .anyRequest().authenticated()
                .and()
            .httpBasic(); // Включаем Basic-аутентификацию
        return http.build();
    }
}

<a id="q4"></a>4. Как мы можем добавить секьюрность к контроллеру? (минимум 2 способа)
Есть несколько способов, вот основные:

Способ 1: Глобальная конфигурация через HttpSecurity
java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeRequests()
                .antMatchers("/public/**").permitAll()  // Доступно всем
                .antMatchers("/users/**").hasRole("USER")  // Только USER
                .antMatchers("/admin/**").hasRole("ADMIN")  // Только ADMIN
                .anyRequest().authenticated()  // Все остальные запросы требуют аутентификации
                .and()
            .httpBasic();
        
        return http.build();
    }
}
Способ 2: Аннотации на уровне методов
java
@RestController
@RequestMapping("/api/users")
public class UserController {
    
    @GetMapping("/profile")
    @PreAuthorize("isAuthenticated()")  // Только аутентифицированные
    public User getProfile() {
        return userService.getCurrentUser();
    }
    
    @GetMapping
    @PreAuthorize("hasRole('ADMIN')")  // Только ADMIN
    public List<User> getAllUsers() {
        return userService.findAll();
    }
    
    @PostMapping
    @Secured("ROLE_ADMIN")  // Альтернативная аннотация
    public User createUser(@RequestBody User user) {
        return userService.save(user);
    }
}
Для работы аннотаций нужно включить:

java
@Configuration
@EnableGlobalMethodSecurity(prePostEnabled = true, securedEnabled = true)
public class MethodSecurityConfig {
    // ...
}

<a id="q5"></a>5. Связи таблиц many-to-many, one-to-one
В JPA/Hibernate эти связи настраиваются аннотациями. Понимание важно и для Hibernate, и для проектирования БД.

One-to-One (Один к одному)
Каждой записи в одной таблице соответствует ровно одна запись в другой.

Пример: пользователь и паспорт

java
@Entity
public class User {
    @Id
    @GeneratedValue
    private Long id;
    
    @OneToOne(mappedBy = "user")
    private Passport passport;
}

@Entity
public class Passport {
    @Id
    @GeneratedValue
    private Long id;
    
    @OneToOne
    @JoinColumn(name = "user_id")
    private User user;
}
В БД: обычно внешний ключ на стороне "владельца" (здесь в passport есть user_id).

Many-to-Many (Многие ко многим)
Например, студенты и курсы: один студент может учиться на многих курсах, на одном курсе много студентов.

Вариант 1: Простая связь через @ManyToMany

java
@Entity
public class Student {
    @Id
    @GeneratedValue
    private Long id;
    
    @ManyToMany
    @JoinTable(
        name = "student_course",
        joinColumns = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private List<Course> courses;
}

@Entity
public class Course {
    @Id
    @GeneratedValue
    private Long id;
    
    @ManyToMany(mappedBy = "courses")
    private List<Student> students;
}

<a id="q6"></a>6. Как работают каскады для таблиц и какие они бывают?
Каскады определяют, как операции над родительской сущностью влияют на связанные дочерние.

Типы каскадов в JPA/Hibernate:
Тип	Описание
ALL	Все операции каскадируются (persist, merge, remove, refresh, detach)
PERSIST	При сохранении родителя сохраняются и дети
MERGE	При обновлении родителя обновляются и дети
REMOVE	При удалении родителя удаляются и дети
REFRESH	При обновлении состояния родителя обновляются и дети
DETACH	При отсоединении родителя отсоединяются и дети
Пример использования:
java
@Entity
public class User {
    @Id
    @GeneratedValue
    private Long id;
    
    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL)
    private List<Order> orders;
}

// Теперь при сохранении User:
User user = new User();
Order order = new Order();
user.setOrders(List.of(order));

userRepository.save(user);  // order сохранится автоматически!

<a id="q8"></a>8. REST-сервисы. Их преимущества и недостатки
REST (Representational State Transfer) — архитектурный стиль для создания веб-сервисов, основанный на принципах HTTP.

Преимущества REST :
✅ Простота — использует стандартные HTTP-методы

✅ Масштабируемость — stateless позволяет легко масштабировать

✅ Гибкость — разные форматы данных (JSON, XML, HTML)

✅ Кэширование — повышает производительность

✅ Универсальность — подходит для разных клиентов (web, mobile, IoT)

Недостатки и ограничения :
❌ Избыточность данных — часто возвращает больше данных, чем нужно

❌ Сложность с связанными данными — может потребоваться несколько запросов

❌ Отсутствие строгой спецификации — многое остается на усмотрение разработчика

<a id="q9"></a>9. Форматы данных, использующиеся в REST-сервисах
Основные форматы обмена данными в REST:

Формат	Особенности	Пример
JSON	Самый популярный, легкий, человекочитаемый	{"name": "Иван", "age": 25}

<a id="q10"></a>10. Что такое responseBody, requestBody, ResponseEntity?
Это аннотации и классы Spring для работы с HTTP.

@RequestBody
Что делает: преобразует тело HTTP-запроса в Java-объект

Где ставить: на параметр метода контроллера

java
@PostMapping("/users")
public User createUser(@RequestBody User user) {
    return userService.save(user);
}
@ResponseBody
Что делает: преобразует Java-объект в тело HTTP-ответа

Где ставить: на метод контроллера или класс

java
@GetMapping("/users/{id}")
@ResponseBody
public User getUser(@PathVariable Long id) {
    return userService.findById(id);
}
ResponseEntity
Что это: полное управление HTTP-ответом (тело, статус, заголовки)

java
@GetMapping("/users/{id}")
public ResponseEntity<User> getUser(@PathVariable Long id) {
    User user = userService.findById(id);
    if (user == null) {
        return ResponseEntity.notFound().build();
    }
    return ResponseEntity.ok(user);
}

<a id="q11"></a>11. Что такое AJAX/fetch?
AJAX (Asynchronous JavaScript and XML)
Технология для асинхронного обмена данными с сервером

Позволяет обновлять часть страницы без перезагрузки

Fetch API
Современная замена AJAX (XMLHttpRequest)

Встроена в современные браузеры

Использует Promise, более удобный синтаксис

Пример fetch :
javascript
// GET-запрос
fetch('https://api.example.com/users/1')
    .then(response => response.json())
    .then(data => console.log(data))
    .catch(error => console.error('Ошибка:', error));

// POST-запрос с JSON
fetch('https://api.example.com/users', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json'
    },
    body: JSON.stringify({name: 'Иван', age: 25})
})
.then(response => response.json())
.then(data => console.log(data));

<a id="q12"></a>12. Чем аннотация RestController отличается от Controller?
Это важное различие в Spring MVC.

Аннотация	@Controller	@RestController
Возвращаемое значение	Имя представления (view)	Объект → JSON/XML
Использование	Веб-страницы (HTML, JSP, Thymeleaf)	REST API
Нужна ли @ResponseBody	Да, на каждом методе	Нет, автоматически
Пример @Controller:
java
@Controller
public class WebController {
    
    @GetMapping("/hello")
    public String hello(Model model) {
        model.addAttribute("message", "Hello!");
        return "hello";
    }
}
Пример @RestController :
java
@RestController
public class ApiController {
    
    @GetMapping("/api/hello")
    public Message hello() {
        return new Message("Hello!");
    }
}

<a id="q13"></a>13. RestTemplate и его методы
RestTemplate — синхронный HTTP-клиент в Spring для вызова REST-сервисов.

Основные методы RestTemplate:
Метод	Описание	HTTP-метод
getForObject()	GET-запрос, возвращает объект	GET
getForEntity()	GET-запрос, возвращает ResponseEntity	GET
postForObject()	POST-запрос, возвращает объект	POST
postForEntity()	POST-запрос, возвращает ResponseEntity	POST
put()	PUT-запрос	PUT
delete()	DELETE-запрос	DELETE
exchange()	Полный контроль над запросом	Любой
Пример использования:
java
@Service
public class UserServiceClient {
    
    private final RestTemplate restTemplate;
    
    public UserServiceClient() {
        this.restTemplate = new RestTemplate();
    }
    
    public User getUser(Long id) {
        String url = "https://api.example.com/users/{id}";
        return restTemplate.getForObject(url, User.class, id);
    }
}

<a id="q14"></a>14. HTTP протокол
HTTP (Hypertext Transfer Protocol) — протокол прикладного уровня для передачи гипертекста.

Основные свойства :
Клиент-серверная архитектура — клиенты инициируют запросы, серверы отвечают

Stateless (без состояния) — каждый запрос независим

Методы HTTP-запросов :
Метод	Описание
GET	Получение ресурса
POST	Создание ресурса
PUT	Полное обновление ресурса
PATCH	Частичное обновление
DELETE	Удаление ресурса
Коды ответов HTTP :
Код	Значение
200 OK	Успешный запрос
201 Created	Ресурс создан
400 Bad Request	Ошибка клиента
401 Unauthorized	Не авторизован
403 Forbidden	Доступ запрещен
404 Not Found	Ресурс не найден
500 Internal Server Error	Ошибка сервера
