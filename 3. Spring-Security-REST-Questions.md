## 4. Чем отличается InMemoryAuthentication от basicAuthentication? [вопросы](#вопросы)
📌 1. Что такое InMemoryAuthentication

InMemoryAuthentication — это способ хранения пользователей в памяти приложения (без базы данных).

👉 Это НЕ способ аутентификации, а источник пользователей (UserDetailsService).

🔎 Как работает

Spring хранит логины/пароли в RAM

При логине сравнивает введённые данные с теми, что в памяти

🧑‍💻 Пример
@Bean
public UserDetailsService users() {
    UserDetails user = User
        .withUsername("user")
        .password("{noop}123")
        .roles("USER")
        .build();

    return new InMemoryUserDetailsManager(user);
}
📌 Где используется
✅ 1. Тестирование

unit/integration тесты

✅ 2. Demo / pet-проекты

быстрый старт без БД

✅ 3. Прототипы

когда не нужна реальная база пользователей

❌ Где НЕ используется

production ❌

реальные системы ❌

👉 потому что:

данные исчезают при перезапуске

нельзя масштабировать

⚠️ Подводные камни

пароли нужно кодировать ({noop} только для теста)

нет персистентности

не подходит для real users

💬 Вопросы
❓ Это способ аутентификации?

❌ Нет → это storage пользователей

❓ Можно ли использовать с BasicAuth?

✔ Да

📌 2. Что такое BasicAuthentication

BasicAuthentication — это способ аутентификации через HTTP заголовок:

Authorization: Basic base64(login:password)

👉 Это уже механизм аутентификации, а не storage

🔎 Как работает

Клиент отправляет login/password в header

Spring декодирует base64

Проверяет через UserDetailsService (например InMemory или БД)

🧑‍💻 Пример
http
    .authorizeHttpRequests(auth -> auth
        .anyRequest().authenticated()
    )
    .httpBasic();
📌 Где используется
✅ 1. Внутренние API

сервис → сервис

✅ 2. Простые REST API

admin endpoints

тестовые API

✅ 3. Интеграции

legacy системы

❌ Где НЕ используется

публичные API ❌

сложные системы ❌

👉 используют JWT/OAuth2

⚠️ Подводные камни

пароль передаётся каждый запрос

base64 ≠ шифрование

нужен HTTPS

💬 Вопросы
❓ Stateless ли BasicAuth?

✔ Да

❓ Почему небезопасен?

✔ пароль передаётся всегда

🔥 3. Ключевая разница
	InMemoryAuthentication	BasicAuthentication
Что это	storage пользователей	механизм аутентификации
Где хранит данные	в памяти	не хранит
Stateless	❌	✔
Используется с	Basic/JWT/FormLogin	UserDetailsService
Production	❌	⚠️ редко
🧑‍💻 4. Как они работают вместе (ВАЖНО)

👉 Очень любят на собесе

🔁 Связка:

BasicAuth → получает логин/пароль

InMemory → хранит пользователей

Пример
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .anyRequest().authenticated()
        )
        .httpBasic(); // механизм

    return http.build();
}
@Bean
public UserDetailsService users() {
    return new InMemoryUserDetailsManager(
        User.withUsername("user")
            .password("{noop}123")
            .roles("USER")
            .build()
    );
}

👉 Что происходит:

клиент отправляет BasicAuth header

Spring берёт пользователя из InMemory

сравнивает пароль

⚠️ Главная ошибка новичков

❌ думают что:

InMemory = способ логина

BasicAuth = хранение

✔ на самом деле:

InMemory = где хранятся пользователи

BasicAuth = как они логинятся

💬 Вопросы от сеньора
❓ Можно ли использовать BasicAuth без InMemory?

✔ Да → с БД, LDAP, кастомным UserDetailsService

❓ Что будет если нет UserDetailsService?

✔ аутентификация не сработает

❓ Что происходит при запросе?

✔ ответ:

берётся header

декодируется

создаётся Authentication

проверяется через UserDetailsService

❓ Почему BasicAuth stateless?

✔ сервер не хранит сессию

❓ Что лучше в production?

✔ JWT / OAuth2

🧠 Как ответить на собесе (идеально)

InMemoryAuthentication — это способ хранения пользователей в памяти, обычно используется для тестов или демо.
BasicAuthentication — это механизм аутентификации через HTTP header.
Обычно они используются вместе: BasicAuth получает логин и пароль, а InMemory хранит пользователей. В production чаще используют БД + JWT вместо этого.

1. Что такое Session (Stateful authentication)

Session-based аутентификация — это способ, при котором сервер хранит состояние пользователя.

🔎 Как работает

Пользователь логинится (login/password)

Сервер создаёт Session

Возвращает клиенту Session ID (cookie)

Клиент отправляет cookie в каждом запросе

Сервер по Session ID находит пользователя

🧑‍💻 Пример
Cookie: JSESSIONID=abc123
📌 Где используется
✅ 1. Классические веб-приложения

Spring MVC

JSP/Thymeleaf

✅ 2. Админ-панели

где есть браузер и cookies

⚠️ Подводные камни

хранит состояние → плохо масштабируется

нужен sticky session (или Redis)

CSRF атаки

💬 Вопросы
❓ Почему это stateful?

✔ сервер хранит состояние

❓ Где хранится сессия?

✔ в памяти / Redis / DB

📌 2. Что такое JWT (Stateless authentication)

JWT (JSON Web Token) — это токен, который хранит данные пользователя на стороне клиента.

🔎 Как работает

Пользователь логинится

Сервер генерирует JWT

Отдаёт клиенту

Клиент отправляет JWT в header:

Authorization: Bearer <token>

Сервер проверяет подпись токена

🧑‍💻 Пример JWT (структура)
header.payload.signature
📌 Где используется
✅ 1. REST API

мобильные приложения

SPA (React, Angular)

✅ 2. Микросервисы

сервис → сервис

⚠️ Подводные камни

нельзя “разлогинить” просто так

нужен refresh token

токен может быть украден

💬 Вопросы
❓ Почему JWT stateless?

✔ сервер не хранит состояние

❓ Где хранится JWT?

✔ client (localStorage / cookie)

🔥 3. Ключевая разница
	Session	JWT
хранение	сервер	клиент
state	stateful	stateless
масштабируемость	❌	✔
logout	легко	сложно
безопасность	CSRF	XSS
🧠 4. Когда использовать
Session

✔ классические веб-приложения
✔ серверный рендеринг

JWT

✔ REST API
✔ мобильные приложения
✔ микросервисы

🧑‍💻 5. Как это выглядит в Spring

    .formLogin();
JWT (упрощённо)

    .sessionManagement()

⚠️ Главные ошибки новичков

❌ JWT всегда лучше
✔ нет → зависит от задачи

❌ JWT безопаснее
✔ нет → просто другой подход

💬 Вопросы от сеньора
❓ Почему JWT лучше масштабируется?

✔ сервер не хранит состояние

❓ Как сделать logout с JWT?

✔ blacklist / short TTL / refresh token

❓ Где хранить JWT?

✔ httpOnly cookie (лучше чем localStorage)

❓ Что безопаснее — JWT или Session?

✔ зависит:

Session → CSRF

JWT → XSS

❓ Можно ли использовать JWT + Session?

✔ Да (редко, но возможно)

🧠 Как ответить на собесе (идеально)

Session-based аутентификация хранит состояние на сервере и подходит для классических веб-приложений.
JWT — это stateless подход, где данные пользователя хранятся в токене на клиенте.
JWT лучше подходит для REST API и микросервисов, потому что масштабируется легче.


## 5. Как добавить секьюрность к контроллеру? (2 способа) [вопросы](#вопросы)
В Spring Security защитить контроллер можно минимум двумя основными способами:

Через конфигурацию безопасности

Через аннотации на методах или классе

На собеседовании важно не просто назвать способы, а объяснить:

на каком уровне работает защита

когда какой способ лучше применять

как они работают внутри

в чём их плюсы и минусы

✅ 1. Защита через конфигурацию Spring Security

Это способ, при котором доступ к URL настраивается в конфигурации безопасности.

То есть мы говорим Spring Security:

к каким URL можно обращаться всем

к каким только авторизованным

к каким только с определённой ролью

📌 Где обычно настраивается

Обычно в SecurityFilterChain.

Пример:

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/public/**").permitAll()
            .requestMatchers("/user/**").hasRole("USER")
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .anyRequest().authenticated()
        )
        .formLogin();

    return http.build();
}
🔎 Как это работает

Когда приходит HTTP-запрос:

Он проходит через SecurityFilterChain

Spring смотрит на URL запроса

Сравнивает его с requestMatchers(...)

Проверяет:

пользователь аутентифицирован или нет

есть ли нужная роль

Если условие подходит — доступ разрешается

Если нет — возвращается 401 или 403

🧠 Что означают основные методы
permitAll()

Разрешает доступ всем, даже неавторизованным пользователям.

Пример:

.requestMatchers("/login", "/register").permitAll()

Используется для:

страницы логина

регистрации

публичного API

swagger/doc endpoints

authenticated()

Требует только факт входа в систему.

.anyRequest().authenticated()

То есть:

пользователь должен быть залогинен

конкретная роль не требуется

hasRole("ADMIN")

Требует роль ADMIN.

.requestMatchers("/admin/**").hasRole("ADMIN")

Важно:
Spring внутри обычно ожидает authority вида:

ROLE_ADMIN

То есть hasRole("ADMIN") фактически проверяет ROLE_ADMIN.

hasAuthority("ROLE_ADMIN")

Проверяет authority напрямую.

.requestMatchers("/admin/**").hasAuthority("ROLE_ADMIN")

Разница:

hasRole("ADMIN") сам добавит ROLE_

hasAuthority("ROLE_ADMIN") ничего не добавляет

hasAnyRole("ADMIN", "MANAGER")

Проверяет, есть ли хотя бы одна из ролей.

.requestMatchers("/management/**").hasAnyRole("ADMIN", "MANAGER")
🧑‍💻 Пример контроллера
@RestController
@RequestMapping("/admin")
public class AdminController {

    @GetMapping("/dashboard")
    public String dashboard() {
        return "Admin dashboard";
    }
}

И конфигурация:

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .anyRequest().authenticated()
        )
        .httpBasic();

    return http.build();
}
📌 Где используется такой способ

Используется, когда нужно защищать доступ по URL:

/admin/** — только ADMIN

/user/** — USER и ADMIN

/public/** — всем

Это удобно для:

REST API

веб-приложений

глобальных правил доступа

✅ Плюсы

централизованная настройка

видно всю политику безопасности в одном месте

удобно ограничивать по URL

легко читать архитектуру доступа

❌ Минусы

плохо подходит для сложной бизнес-логики

не умеет удобно проверять параметры метода

иногда URL-защиты недостаточно

⚠️ Подводные камни
1. Порядок правил важен

Spring применяет правила сверху вниз.

Если сначала написать:

.anyRequest().authenticated()

а потом:

.requestMatchers("/admin/**").hasRole("ADMIN")

то второе правило уже может не сработать как ожидается.

2. Ошибка с ROLE_

Очень частая ошибка:

.hasRole("ROLE_ADMIN")

Это неправильно, потому что hasRole() сам добавит ROLE_.

Правильно:

.hasRole("ADMIN")
3. 401 и 403

Нужно понимать разницу:

401 Unauthorized — пользователь не аутентифицирован

403 Forbidden — пользователь залогинен, но прав нет

Сеньоры очень любят это спрашивать.

💬 Вопросы от сеньора
❓ Когда лучше защищать через конфиг, а не через аннотации?

✔ Когда правила завязаны на URL и должны быть централизованными.

❓ Почему этот способ удобен?

✔ Вся политика доступа в одном месте.

❓ Что произойдёт при запросе к /admin от обычного USER?

✔ Если пользователь аутентифицирован, но не ADMIN — вернётся 403 Forbidden.

✅ 2. Защита через аннотации на методах и классах

Это называется method security.

Мы можем повесить ограничения:

на весь контроллер

на конкретный метод контроллера

даже на метод сервиса

Это более гибкий подход, потому что можно проверять не только роль, но и условия.

📌 Чтобы это работало

Нужно включить method security:

@Configuration
@EnableMethodSecurity
public class SecurityConfig {
}

Без этого @PreAuthorize, @Secured и другие аннотации работать не будут.

Основные аннотации
@PreAuthorize

Проверяет условие до выполнения метода.

@Secured

Проверяет роли.

@RolesAllowed

Похож на @Secured, но из JSR-250.

✅ Вариант 2.1 — @PreAuthorize

Это самый гибкий и часто рекомендуемый способ.

Пример:

@RestController
@RequestMapping("/admin")
public class AdminController {

    @PreAuthorize("hasRole('ADMIN')")
    @GetMapping("/dashboard")
    public String dashboard() {
        return "Admin dashboard";
    }
}
🔎 Как это работает

Перед вызовом метода Spring Security:

Перехватывает вызов

Читает выражение в @PreAuthorize

Проверяет его

Если условие истинно — метод выполняется

Если ложно — выбрасывается ошибка доступа (403)

📌 Что можно писать в @PreAuthorize
Проверка роли
@PreAuthorize("hasRole('ADMIN')")
Проверка одной из ролей
@PreAuthorize("hasAnyRole('ADMIN', 'MANAGER')")
Проверка authority
@PreAuthorize("hasAuthority('READ_PRIVILEGE')")
Проверка текущего пользователя и параметра метода
@PreAuthorize("#id == authentication.principal.id")

Это значит:

метод можно вызвать только если id совпадает с id текущего пользователя

Проверка через bean
@PreAuthorize("@userSecurityService.canAccessUser(#id, authentication.name)")

Это уже уровень middle/senior.

Здесь вызывается кастомный bean, который сам решает, есть доступ или нет.

🧑‍💻 Пример
@RestController
@RequestMapping("/users")
public class UserController {

    @PreAuthorize("hasRole('ADMIN') or #id == authentication.principal.id")
    @GetMapping("/{id}")
    public String getUser(@PathVariable Long id) {
        return "User data";
    }
}
Где используется @PreAuthorize

когда нужна защита на уровне метода

когда недостаточно проверки только по URL

когда важны параметры запроса

когда нужен доступ “только к своим данным”

Примеры:

пользователь может смотреть только свой профиль

админ может смотреть все профили

менеджер может редактировать только своих клиентов

✅ Плюсы

очень гибкий

можно использовать SpEL

удобно проверять параметры метода

можно строить сложную бизнес-логику доступа

❌ Минусы

логика безопасности размазывается по коду

сложные выражения трудно читать

можно переборщить с количеством условий

⚠️ Подводные камни
1. Нужен @EnableMethodSecurity

Без него аннотации просто не будут работать.

2. Ошибка с ролями

Правильно:

hasRole('ADMIN')

а не:

hasRole('ROLE_ADMIN')
3. Слишком сложные SpEL-выражения

Если выражение очень длинное, код становится плохо читаемым.
Лучше вынести в отдельный сервис.

💬 Вопросы от сеньора
❓ Почему @PreAuthorize считается мощнее @Secured?

✔ Потому что поддерживает SpEL и сложные условия.

❓ Можно ли проверять аргументы метода?

✔ Да, через #parameterName.

❓ Когда лучше использовать @PreAuthorize, а не URL-конфиг?

✔ Когда доступ зависит не только от URL, но и от бизнес-логики и параметров метода.

✅ Вариант 2.2 — @Secured

Это более простой вариант method security.

Пример:

@RestController
@RequestMapping("/admin")
public class AdminController {

    @Secured("ROLE_ADMIN")
    @GetMapping("/dashboard")
    public String dashboard() {
        return "Admin dashboard";
    }
}
📌 Как работает

Перед вызовом метода проверяется, есть ли у пользователя указанная роль.

Особенность

В отличие от hasRole("ADMIN"), здесь обычно пишут именно:

ROLE_ADMIN

То есть @Secured не добавляет ROLE_ автоматически.

Где используется

когда нужна простая проверка ролей

когда не нужна сложная логика

в старых проектах

✅ Плюсы

простой

легко читается

подходит для базовой защиты

❌ Минусы

не поддерживает SpEL

нельзя проверять параметры метода

менее гибкий, чем @PreAuthorize

💬 Вопросы
❓ Когда @Secured хуже @PreAuthorize?

✔ Когда нужны сложные условия доступа.

❓ Почему @PreAuthorize чаще выбирают?

✔ Он гибче.

✅ Можно ли вешать аннотацию на класс?

Да.

Пример:

@RestController
@RequestMapping("/admin")
@PreAuthorize("hasRole('ADMIN')")
public class AdminController {

    @GetMapping("/dashboard")
    public String dashboard() {
        return "Admin dashboard";
    }

    @GetMapping("/stats")
    public String stats() {
        return "Stats";
    }
}

Тогда правило действует на все методы класса.

Где это удобно

Когда весь контроллер должен быть доступен только одной роли.

Подводный камень

Если один метод должен быть доступен иначе, нужно быть аккуратным с переопределением.

🔥 Что любит спрашивать сеньор дополнительно
❓ Что лучше: защищать контроллер или сервис?

Хороший ответ:

Лучше защищать не только контроллер, но и сервисный слой, потому что сервис может быть вызван не только из контроллера. Контроллерная защита — это внешний уровень, а сервисная — более надёжный уровень бизнес-правил.

Это очень сильный ответ.

❓ Почему недостаточно защищать только URL?

✔ Потому что один и тот же URL может требовать дополнительных условий, например доступ только к своим данным.

❓ Что произойдёт, если убрать @EnableMethodSecurity?

✔ @PreAuthorize и @Secured не будут работать.

❓ Можно ли использовать @PreAuthorize в сервисе?

✔ Да, и это часто даже правильнее.

Пример:

@Service
public class UserService {

    @PreAuthorize("#id == authentication.principal.id or hasRole('ADMIN')")
    public String getUserData(Long id) {
        return "data";
    }
}
❓ Что будет, если пользователь не залогинен?

✔ Обычно 401 Unauthorized

❓ А если залогинен, но прав нет?

✔ 403 Forbidden

🧠 Как ответить на собеседовании идеально

Добавить секьюрность к контроллеру можно минимум двумя способами.
Первый — через конфигурацию Spring Security, например в SecurityFilterChain, где доступ ограничивается по URL через requestMatchers, authenticated, hasRole и так далее. Это удобно для глобальных правил доступа.
Второй — через method security, например @PreAuthorize или @Secured, когда ограничения ставятся на методы или классы. @PreAuthorize более гибкий, потому что поддерживает SpEL и позволяет проверять не только роли, но и параметры метода.
На практике часто используют оба подхода вместе: базовые правила — через конфиг, сложную бизнес-логику — через аннотации, часто даже на сервисном слое.

## 6. Связи таблиц: @ManyToMany и @OneToOne [вопросы](#вопросы)
В JPA (Hibernate) связи между таблицами описывают, как объекты связаны друг с другом в базе данных.

Самые важные типы связей:

One-to-One (один к одному)

Many-to-Many (многие ко многим)

📌 1. One-to-One (один к одному)
📖 Что это

One-to-One — это связь, при которой одной записи в таблице соответствует ровно одна запись в другой таблице.

👉 Проще:

один пользователь → один профиль

один человек → один паспорт

🧑‍💻 Пример из жизни

User → Profile

Person → Passport

Order → Invoice

🧑‍💻 Пример в JPA
@Entity
public class User {

    @Id
    @GeneratedValue
    private Long id;

    @OneToOne
    @JoinColumn(name = "profile_id")
    private Profile profile;
}
@Entity
public class Profile {

    @Id
    @GeneratedValue
    private Long id;

    private String bio;
}
🔎 Что здесь происходит

@OneToOne — говорит о связи 1 к 1

@JoinColumn — указывает внешний ключ (profile_id)

таблица User содержит ссылку на Profile

📌 Где хранится внешний ключ

👉 В таблице, где стоит @JoinColumn

В данном случае:

User.profile_id → Profile.id

🔁 Двусторонняя связь
@Entity
public class Profile {

    @OneToOne(mappedBy = "profile")
    private User user;
}
🔎 Что такое mappedBy

👉 означает:

эта сторона не владеет связью

внешний ключ хранится в другой таблице

📌 Owning side

👉 сторона без mappedBy
👉 именно она управляет связью

⚠️ Подводные камни
1. Где хранить FK?

Очень важный вопрос:

👉 внешний ключ должен быть только в одной таблице

2. Lazy loading

По умолчанию:

OneToOne → EAGER (может грузить лишние данные)

👉 лучше явно указывать:

@OneToOne(fetch = FetchType.LAZY)
3. nullable

Если связь обязательна:

@JoinColumn(nullable = false)
4. Частая ошибка

Забывают mappedBy → создаётся лишняя колонка

💬 Вопросы от сеньора
❓ Где хранится внешний ключ?

✔ в owning side (@JoinColumn)

❓ Что такое owning side?

✔ сторона без mappedBy

❓ Почему OneToOne может быть дорогим?

✔ EAGER loading → лишние запросы

❓ Когда использовать OneToOne?

✔ когда сущность логически 1 к 1 (Profile, Passport)

📌 2. Many-to-Many (многие ко многим)
📖 Что это

Many-to-Many — это связь, при которой:

одна запись связана с множеством других

и наоборот

👉 пример:

пользователь имеет много ролей

роль принадлежит многим пользователям

🧑‍💻 Пример из жизни

User ↔ Role

Student ↔ Course

Product ↔ Category

🔎 Как реализуется в БД

👉 через промежуточную таблицу (join table)

Пример:

users
roles
users_roles   ← промежуточная таблица
🧑‍💻 Пример в JPA
@Entity
public class User {

    @Id
    @GeneratedValue
    private Long id;

    @ManyToMany
    @JoinTable(
        name = "users_roles",
        joinColumns = @JoinColumn(name = "user_id"),
        inverseJoinColumns = @JoinColumn(name = "role_id")
    )
    private List<Role> roles;
}
@Entity
public class Role {

    @Id
    @GeneratedValue
    private Long id;

    private String name;
}
🔎 Что здесь происходит

@ManyToMany — связь многие ко многим

@JoinTable — задаёт промежуточную таблицу

joinColumns — FK на текущую сущность

inverseJoinColumns — FK на связанную сущность

🔁 Двусторонняя связь
@Entity
public class Role {

    @ManyToMany(mappedBy = "roles")
    private List<User> users;
}
📌 Owning side

👉 сторона, где есть @JoinTable

⚠️ Подводные камни (очень важно)
1. Lazy loading

По умолчанию:

@ManyToMany(fetch = FetchType.LAZY)

👉 это хорошо, но может привести к:

❗ N+1 проблеме
2. N+1 проблема

👉 при загрузке списка пользователей:

сначала 1 запрос users

потом N запросов roles

3. Cascade опасен
cascade = CascadeType.ALL

👉 может удалить роли при удалении пользователя ❌

4. Плохая практика

👉 ManyToMany часто не используют напрямую
✔ лучше делать промежуточную сущность

🧠 Правильный подход (очень любят на собесе)

Вместо:

User ↔ Role

делают:

User → UserRole → Role
🧑‍💻 Пример
@Entity
public class UserRole {

    @ManyToOne
    private User user;

    @ManyToOne
    private Role role;
}
📌 Почему так лучше

можно добавить поля (например дата назначения роли)

больше контроля

проще масштабировать

💬 Вопросы от сеньора
❓ Как реализуется many-to-many в БД?

✔ через join table

❓ Где хранится связь?

✔ в промежуточной таблице

❓ Почему ManyToMany — плохая практика?

✔ сложно управлять + нет гибкости

❓ Как правильно сделать?

✔ через промежуточную сущность

❓ Что такое N+1 проблема?

✔ много лишних запросов к БД

🔥 Ключевая разница
	One-to-One	Many-to-Many
связь	1 к 1	многие ко многим
FK	в одной таблице	через join table
сложность	простая	сложная
частые проблемы	eager loading	N+1, cascade
🧠 Как ответить на собесе (идеально)

В JPA связи One-to-One используются, когда одна сущность связана строго с одной другой, например User и Profile. Внешний ключ хранится в owning side, которая определяется через @JoinColumn.
Many-to-Many используется, когда сущности связаны множественно, например User и Role. В базе это реализуется через промежуточную таблицу.
На практике Many-to-Many часто заменяют на две связи One-to-Many через промежуточную сущность, чтобы иметь больше контроля и избежать проблем с каскадами и расширяемостью.

📌 Короткая выжимка
OneToOne

@JoinColumn(name = "profile_id")

🧠 Как ответить на собесе (идеально)

HTTP — это протокол клиент-серверного взаимодействия, при котором клиент отправляет запрос, а сервер возвращает ответ. Он stateless, то есть сервер не хранит состояние клиента. Запрос состоит из метода, URL, заголовков и тела, а ответ — из статуса, заголовков и тела. Важно понимать HTTP методы, статус-коды и такие вещи как идемпотентность и разницу между 401 и 403.
